# MagicSquare — `.cursorrules` 명세 및 AI 가이드라인 정리

> Cursor 에디터용 프로젝트 규칙 파일(`.cursorrules`) 설계·작성·검토 과정 보고

---

## 1. 목적

MagicSquare 저장소에서 **Python · ECB · TDD · pytest** 방향을 에디터 AI와 공유하기 위해, YAML 형식의 `.cursorrules`를 두었다. 본 문서는 해당 파일을 **어떻게 만들었고**, **무엇을 담았으며**, **검토 시 어떤 점을 확인했는지**를 정리한다.

---

## 2. 작업 단계 요약

| 단계 | 내용 |
|------|------|
| 1 | 최상위 섹션만 포함한 YAML 뼈대 작성 (`project`, `code_style`, `architecture`, `tdd_rules`, `testing`, `forbidden`, `file_structure`, `ai_behavior`). 각 섹션 앞 80자 `#` 구분선. |
| 2 | `tdd_rules`만 우선 채움: red / green / refactor를 한 줄 요약 문자열로 기술. |
| 3 | 검토: YAML 문법, 필수 섹션 존재, `tdd_rules`와 `forbidden` 충돌, `ai_behavior` 실행 가능성 — **당시** 빈 섹션이 많았고 `forbidden`이 비어 충돌은 없음으로 판단. |
| 4 | 모든 빈 섹션을 MagicSquare 맥락에 맞게 채움. `tdd_rules`는 phase별 `description` · `rules` · `must_not` 구조로 세분화. |

---

## 3. 산출물 위치 및 형식

- **파일 경로**: 저장소 루트 `.cursorrules`
- **형식**: YAML (주석 구분선 `################################################################################` 유지)
- **파서 검증**: Python `yaml.safe_load`로 로드 가능함을 확인했으며, 문자열 내 따옴표·콜론(`except:` 등)은 따옴표로 이스케이프했다.

---

## 4. 섹션별 요지

### 4.1 `project`

프로젝트명 MagicSquare, 4×4 마방진 도메인, `Prompting/`·`Report/`와의 정렬, ECB·TDD 목표를 명시했다.

### 4.2 `code_style`

요청사항대로 Python 3.10+, PEP 8, 전 함수 타입 힌트, Google 스타일 docstring(public 메서드), 줄 길이 88(Black 기준)을 기술했다.

### 4.3 `architecture`

ECB 세 레이어를 정의했다.

- **boundary**: UI, API, CLI 등 외부 입출력.
- **control**: 유스케이스·마방진 검증·생성 흐름 조정.
- **entity**: Grid·Cell·합계 검증 등 순수 도메인.

의존성 방향: **boundary → control → entity** 한 방향. entity는 상위 레이어를 참조하지 않는다.

### 4.4 `tdd_rules`

각 phase(red / green / refactor)마다 **설명**, **지켜야 할 규칙 목록**, **금지 사항**을 분리했다. green 단계에서 리팩터링 금지, refactor 단계에서 커버리지 하한 유지 등이 포함된다.

### 4.5 `testing`

pytest, AAA, 최소 커버리지 80%, fixture 스코프(function / module / session) 규칙, 파일·함수 네이밍(`test_` 접두사)을 명시했다.

### 4.6 `forbidden`

항목마다 `pattern` · `reason` · `alternative` 구조로, 최소 요구인 **print() 디버깅**, **하드코딩 상수**, **bare/과도한 except** 를 포함했다.

### 4.7 `file_structure`

ECB에 맞춘 `magic_square/` 트리(boundary, control, entity, tests)를 블록 문자열로 기술하고, 루트의 `Prompting/`, `Report/` 등을 언급했다.

### 4.8 `ai_behavior`

코드 생성 **전**·**중**·**후**에 AI가 따를 지침과, TDD/ECB/타입 위반 시 **한국어 경고**를 출력하라는 `on_tdd_violation` 항목을 두었다.

---

## 5. 검토 시 확인한 사항 (요약)

- **YAML**: 구조·들여쓰기·특수 문자 이스케이프 후 문법상 문제 없음.
- **필수 키**: 요청한 최상위 섹션은 모두 존재; 전체 완성 후에는 빈 섹션이 아닌 **내용이 채워진** 상태다.
- **`tdd_rules` vs `forbidden`**: `forbidden`에 TDD와 정면으로 모순되는 문구는 넣지 않았다.
- **AI 실행 한계**: `.cursorrules`는 에디터가 **자연어/구조화 힌트**로 읽는 경우가 많아, “항상 CI 통과까지 자동 수행” 같은 **환경에 묶인 보장**은 이 파일만으로는 완전히 강제되지 않을 수 있다. `ai_behavior`는 그 범위 안에서 **가능한 지시**로 작성했다.

## 6. 향후 참고

- 실제 패키지 디렉터리(`magic_square/`)가 생기면 `file_structure`와 경로를 맞출 것.
- `naming_convention.files`에서 `test_*.py`와 `*_test.py` 중 **하나로 통일**하는 결정이 나면 `.cursorrules`에 한 줄로 고정해 두는 것이 좋다.
- `pyproject.toml`·`black`·`mypy` 등이 추가되면 `code_style`과 상호 링크하거나 중복을 줄이는 편이 유지보수에 유리하다.
