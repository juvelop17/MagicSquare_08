## 2. 수행 내용 상세

### 2.1 To-Do 작성 시 참조 문서 안내

- **질의:** 작성된 문서 중 무엇을 참조해 할 일 목록을 만들면 좋은가.  
- **정리:** PRD를 **기능·AC·추적 매트릭스**의 중심으로 두고, Epic·스토리 분해는 **Report/4**, 계약·오류·U-/D- ID는 **Report/2**, 품질·TDD·ECB는 **Report/3** 및 **`.cursorrules`**, 문제 배경만 필요 시 **Report/1**. `Prompting/`은 문서 생성용 프롬프트에 가깝고 실행 백로그의 직접 원천으로는 PRD·Report가 우선이라고 안내하였다.

### 2.2 구현용 To-Do 구조 (방법론 적용 산출)

- **요청:** 시니어 아키텍트·TDD 코치 관점에서 PRD·Report/2~4·`.cursorrules`를 반영한 **Magic Square 4×4 완성 시스템** 구현용 To-Do.  
- **적용 방법론:** Concept-to-Code Traceability, Dual-Track(UI·Logic) TDD, To-Do → Scenario → Test → Code, ECB, RED–GREEN–REFACTOR.  
- **식별자:** Epic-001, US-001~006, TASK-001~021 등 (대화 산출물). Phase별로 도메인(보드·빈칸·누락·판정·두 시도)·경계(검증·오류 매핑)·Control·통합·커버리지 게이트를 포함.  
- **보관 위치:** 상세 TASK 표는 **대화 기록**에 전개되어 있다. 저장소에는 [`README.md`](../README.md)의 **구현 To-Do** 절에 Epic / User Story / 세분 항목이 **요약 체크리스트**로 반영되어 있으며, 세부 TASK ID·RED/GREEN/REFACTOR·시나리오 레벨(L0~L3)까지의 전개는 필요 시 대화 산출물을 복사해 이슈 트래커 또는 별도 `docs/` 백로그 파일로 옮길 수 있다.

### 2.3 `README.md` 초안 작성

- 프로젝트 한 줄 소개, **문서 관계(한 줄)** — Report/5와 정합.  
- 문서 맵, 범위, Report/4 정렬 User Story 표, Report/2·PRD §8.1 수준의 **입출력·오류 계약** 표.  
- ECB 요약, 구현 To-Do (Epic·US·작업 세분화), 라이선스 자리.

### 2.4 `README.md` 보강 — 가상 환경·실행·리팩터링

- **가상 환경 기본:** `python -m venv .venv`, PowerShell/cmd/Unix **활성화** 절차, 실행 정책(Windows) 참고, `pip` 업그레이드, `pytest`·`pytest-cov` 설치, 향후 `requirements*.txt` / `pyproject.toml` 도입 시 설치 명령 예시.  
- **테스트:** `python -m pytest`, `--cov=` 예시와 PRD §7·`.cursorrules` 커버리지 설명.  
- **IDE/Cursor:** 인터프리터를 `.venv`로 지정.  
- **TDD·리팩터링:** RED–GREEN–REFACTOR 표, GREEN 단계에서 리팩터 금지, REFACTOR 단계에서 기능 혼합 금지, 커버리지·ECB·Dual-Track(PR §8.3) 유지.
