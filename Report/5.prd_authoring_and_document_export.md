## 2. 입력·출력 계약 고정 요약

PRD 본문에 다음을 **측정 가능한 문장**으로 고정하였다.

| 구분 | 고정 내용 |
|------|-----------|
| **입력** | `int[][]` 형태의 4×4; `0`은 빈칸 **정확히 2개**; 셀 값은 `0` 또는 `1`~`16`; `0` 제외 중복 없음; 첫 빈칸은 **row-major** 스캔 시 최초 `0` |
| **출력** | `int[6]` = `[r1, c1, n1, r2, c2, n2]`; 좌표 **1-index** |
| **시도 규칙** | 시도 1: 작은 누락 수→첫 빈칸, 큰 수→둘째 빈칸; 마방진이면 해당 순서로 반환. 시도 1 실패 시 시도 2(순서 반대); 마방진이면 역순 반영 |
| **Magic Constant** | n=4 → **34** |
| **두 시도 모두 실패** | 오류 코드 **`E_DOMAIN`**, 메시지 **`No valid magic square completion exists for this grid.`** |
| **경계 검증 실패** | `E_SIZE`, `E_EMPTY_COUNT`, `E_VALUE_RANGE`, `E_DUPLICATE` 및 PRD §8.1에 명시한 **문자 단위 고정 메시지** |

## 3. PRD 목차(12개 섹션)와 역할

| § | 제목 | 역할 |
|---|------|------|
| 1 | Executive Summary | 한눈 요약·훈련 목표 역량 |
| 2 | Problem Statement | “생성”이 아닌 **검증 가능한 완성** 관점·계약의 필요성 |
| 3 | Target Users | 학습자·리뷰어·실행 환경(테스트/콘솔) |
| 4 | Scope | In-Scope / Out-of-Scope(UI·DB·N×N 일반화·완전 생성 솔버 배제 등) |
| 5 | Functional Requirements | FR-01~FR-05 템플릿(입력·처리·출력·AC·오류) |
| 6 | Business Rules | BR-01~ 형태의 도메인 규칙 |
| 7 | Non-Functional Requirements | 커버리지(도메인 95%+·경계 85%+), 결정론, **입력 행렬 비변경**, 선택 성능 |
| 8 | Dual-Track TDD Strategy | Track A(경계)·Track B(도메인)·병렬 RED/GREEN/REFACTOR 금지 사항 |
| 9 | Test Plan | 시나리오 ID, 회귀 정책, **시드 A/B/C**, Property 체크 |
| 10 | Architecture Overview | Boundary / Domain / 선택 Control, 의존성 방향 |
| 11 | Risks & Ambiguities | 실패 정책·1-index·row-major 등 **결정 항목** 명시 |
| 12 | Traceability Matrix | Concept/Invariant → BR → FR → AC → Test → Component |

## 5. 추적성(Traceability)

PRD **§12**에 다음 형식의 표를 포함하였다.

`Concept/Invariant → Business Rule → Feature(FR) → Acceptance Criteria → Test Case → Component(논리명)`

구현 시 클래스/파일명을 바꾸더라도 **요구 ↔ 테스트 ↔ 규칙** 추적이 가능하도록 **논리 컴포넌트** 단위로 기술하였다.
