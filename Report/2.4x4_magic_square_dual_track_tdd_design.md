# 4×4 Magic Square — Dual-Track UI + Logic TDD 및 Clean Architecture 설계

> 실습: Cursor AI를 활용한 소프트웨어 개발 과정 — 응용 (설계·계약·테스트·통합 계획만, 구현 코드 없음)  
> 입력/출력 고정: `int[4][4]` (0=빈칸 2개, 0 또는 1~16 비중복) → `int[6]` = `[r1,c1,n1,r2,c2,n2]` (1-index, 작은수→첫빈칸·큰수→둘째빈칸이 마방진이면 그 순서, 아니면 반대)

---

## 1) Logic Layer (Domain Layer) 설계

### 1.1 도메인 개념

| 구분 | 이름 | 책임 (SRP) |
|------|------|------------|
| Entity | `MagicSquareGrid` (또는 동등 개념) | 4×4 셀 상태(값/빈칸)를 보유하고, 그리드 단위 불변조건과의 정합성을 표현한다. |
| Value Object | `CellPosition` | (row, col)을 1-index 기준으로 캡슐화하고, 유효 범위(1~4) 검증 규칙을 둔다. |
| Value Object | `MissingNumberPair` | 누락된 두 정수(n₁, n₂)를 순서 의미와 함께 보유한다(어느 빈칸에 어느 수가 들어가는지와 구분). |
| Domain Service | `MagicSquareValidator` | 행·열·대각선 합이 Magic Constant와 일치하는지 판정한다(완전 채워진 그리드 기준). |
| Domain Service | `PartialGridAnalyzer` | 빈칸 2개 좌표 추출, 이미 배치된 숫자 집합으로부터 누락 두 수 유도. |
| Domain Service | `SolutionOrderDecider` | 두 빈칸에 (n₁, n₂)를 (작은수→첫 빈칸, 큰수→둘째 빈칸)으로 넣었을 때 마방진이면 그 순서, 아니면 반대 순서로 확정한다. |

### 1.2 도메인 불변조건 (Invariants)

| ID | 불변조건 | 검증 시점 |
|----|----------|-----------|
| INV-01 | 그리드 크기는 4×4이다. | 그리드 생성/수락 시 |
| INV-02 | 값은 0 또는 1~16이다. | 셀 단위 |
| INV-03 | 0이 아닌 값은 서로 중복되지 않는다. | 그리드 전체 |
| INV-04 | 0인 셀은 정확히 2개이다. | 입력 수락 시 |
| INV-05 | (완전 채움 후) 각 행 합 = 각 열 합 = 두 대각선 합 = **Magic Constant 34** (1~16 전체 사용 시). | 해 판정 시 |
| INV-06 | 출력 좌표 (r,c)는 1-index이며 1≤r,c≤4이다. | 해 반환 시 |

**Magic Constant:** 정수 1~16을 한 번씩 쓰는 4×4 마방진에서 합은 **34**로 고정한다.

### 1.3 핵심 유스케이스 (도메인 관점)

| UC | 이름 | 설명 |
|----|------|------|
| UC-1 | 빈칸 찾기 | 0인 셀의 (r,c) 두 개를 1-index로 반환한다. |
| UC-2 | 누락 숫자 찾기 | {1..16}에서 그리드에 등장하지 않는 정수 두 개를 집합으로 구한다. |
| UC-3 | 마방진 판정 | 임의의 완전 4×4 그리드에 대해 INV-05를 만족하는지 불리언으로 판정한다. |
| UC-4 | 두 조합 시도 | 빈칸 순서 (A,B)에 대해 (n_small→A, n_large→B) 채운 그리드를 UC-3으로 검사하고, 실패 시 (n_large→A, n_small→B)를 검사한다. 성공한 조합으로 (n₁,n₂) 순서를 확정한다. |

**빈칸 순서 규칙 (검증 가능):** 행 우선, 행 번호 오름차순, 같은 행에서는 열 오름차순. 첫 번째 0 = 첫 빈칸, 두 번째 0 = 둘째 빈칸.

### 1.4 Domain API (내부 계약)

코드는 작성하지 않고 메서드 수준·입·출·실패만 기술한다.

| API | 입력 | 출력 | 실패 조건 |
|-----|------|------|-----------|
| `findEmptyCells(grid)` | 4×4, INV-01~04 만족 가정 또는 사전 검증됨 | 두 `CellPosition` (첫/둘째 빈칸) | INV-04 위반 시 도메인 예외 또는 Result 실패 |
| `findMissingNumbers(grid)` | 동일 | 두 정수의 집합 또는 정렬된 쌍 (구현 선택은 앱 레이어에서 일관되게) | 누락이 2개가 아니면 실패 |
| `isMagicSquareComplete(grid)` | 0 없는 4×4 | true/false | INV-01~03 위반 시 실패 |
| `decidePlacementAndOrder(firstEmpty, secondEmpty, missingPair)` | 빈칸 2개, {a,b} (a < b) | 성공 시 (n₁, n₂) — 작은수→첫 빈칸이 마방진이면 [n₁,n₂]=[a,b], 아니면 [b,a] | 둘 다 마방진이 아니면 실패(도메인 불가) |
| `buildResultRowColNumbers(...)` | 위에서 확정한 좌표·숫자 | `int[6]` = [r1,c1,n1,r2,c2,n2] | 좌표·값 범위 불일치 시 실패 |

**실패의 의미:** “유일한 해부터 두 조합 모두 실패”까지를 도메인이 구분할지는 선택이나, 테스트는 “정상 입력이면 둘 중 하나는 항상 성공”하는 교육용 데이터에 맞출지, “해가 없는 그리드”를 도메인 실패로 둘지 **명시**해야 한다(단위 테스트에서 케이스로 고정).

### 1.5 Domain 단위 테스트 설계 (RED 우선)

| # | 케이스 | 입력 요약 | 기대 | 보호 Invariant |
|---|--------|-----------|------|----------------|
| D-01 | 정상: 알려진 4×4 일부(빈칸 2) | 과제용 고정 seed 그리드 1 | `[r1,c1,n1,r2,c2,n2]`가 명세와 일치 | INV-05, 출력 형식 |
| D-02 | 정상: 빈칸 순서가 열이 다른 경우 | seed 그리드 2 | 첫/둘째 빈칸 탐색 순서 규칙과 일치 | UC-1 순서 |
| D-03 | n_small→첫칸이 성공 | 합의된 부분 격자 | n 순서 = (작, 큰) | UC-4 첫 분기 |
| D-04 | n_small→첫칸이 실패, 반대가 성공 | 별도 seed | n 순서 = (큰, 작) | UC-4 둘째 분기 |
| D-05 | `isMagicSquareComplete` true | 완성 4×4 마방진 | true | INV-05 |
| D-06 | `isMagicSquareComplete` false | 합 틀린 완전 그리드 | false | INV-05 |
| D-07 | 빈칸 개수 0, 1, 3 | 그리드 변형 | `findEmptyCells` 실패 | INV-04 |
| D-08 | 중복 값 | 0 제외 중복 | 입력 거부 또는 전처리 전 실패 | INV-03 |
| D-09 | 범위 밖 값 | 17 또는 -1 | 실패 | INV-02 |
| D-10 | (선택) 해 없음 그리드 | 설계에서 허용 시 | `decidePlacement...` 양쪽 실패 | UC-4 |

## 2) Screen Layer (UI Layer) 설계 (Boundary Layer)

### 2.1 사용자/호출자 관점 시나리오

1. **행렬 입력** — 4×4 정수(0 포함)를 경계로 전달한다.  
2. **검증** — 스키마·불변조건을 만족하는지 판단한다.  
3. **Domain 위임** — 유효하면 해를 요청한다(Mock 가능).  
4. **결과 출력** — 성공 시 `int[6]`, 실패 시 오류 스키마를 반환한다.  
5. (선택) **저장** — Data 레이어에 delegate.

### 2.2 UI 계약 (외부 계약)

**Input schema**

| 필드 | 타입 | 제약 |
|------|------|------|
| `grid` | `int[4][4]` 또는 동등 2D 배열 | INV-01~04 |

**Output schema (성공)**

| 필드 | 타입 | 제약 |
|------|------|------|
| `result` | `int[6]` | `[r1,c1,n1,r2,c2,n2]`, 1-index, 1~16 |

**Error schema**

| 필드 | 타입 | 설명 |
|------|------|------|
| `code` | enum 문자열 (고정 집합) | 아래 Error code 표와 1:1 |
| `message` | string | 2.4의 고정 문구 |

**Error code 목록 (검증 가능)**

| code | 조건 |
|------|------|
| `E_SIZE` | 행/열 수가 4가 아님 |
| `E_EMPTY_COUNT` | 0인 셀 개수 ≠ 2 |
| `E_VALUE_RANGE` | 어떤 셀이 0도 아니고 1~16도 아님 |
| `E_DUPLICATE` | 0 제외 값이 중복 |
| `E_DOMAIN` | 입력은 유효하나 해만을 도메인에서 구할 수 없음(선택) |
| `E_DATA` | 저장소 실패(통합 시) |

### 2.3 UI 레벨 테스트 (Contract-first, RED)

| # | 케이스 | 설정 | 기대 |
|---|--------|------|------|
| U-01 | 잘못된 크기 | 3×4 또는 4×3 | `E_SIZE` + 고정 `message` |
| U-02 | 빈칸 0개 | 0 없음 | `E_EMPTY_COUNT` |
| U-03 | 빈칸 3개 | 0 세 개 | `E_EMPTY_COUNT` |
| U-04 | 값 범위 | 17 또는 -5 | `E_VALUE_RANGE` |
| U-05 | 중복 | 두 칸에 동일한 1~16 | `E_DUPLICATE` |
| U-06 | 반환 포맷 | Mock이 고정 `int[6]` 반환 | boundary가 그대로 전달·직렬화 |
| U-07 | Domain Mock 실패 | Mock이 “해 없음” | `E_DOMAIN` (해당 시) |

Domain은 **Mock**으로: 고정 `int[6]` 또는 예외로 UI만 검증.

### 2.4 UX/출력 규칙 (에러 문구 고정)

| code | `message` (정확히 일치) |
|------|-------------------------|
| `E_SIZE` | `Grid must be 4 by 4.` |
| `E_EMPTY_COUNT` | `There must be exactly two empty cells (value 0).` |
| `E_VALUE_RANGE` | `Each cell must be 0 or an integer from 1 to 16.` |
| `E_DUPLICATE` | `Values from 1 to 16 must not repeat except for 0.` |
| `E_DOMAIN` | `No valid magic square completion exists for this grid.` |
| `E_DATA` | `Storage operation failed.` |

(로케일은 별 계약; 위는 기본 EN 고정.)

## 3) Data Layer 설계

### 3.1 목적

- **학습용**으로 동일 유스케이스를 앱 재시작 후에도 복습하거나, **감사/재현**을 위해 마지막 입력·결과를 보관한다.  
- **범위:** 영속 매체 추상화만; 비즈니스 규칙은 Domain에 둔다.

### 3.2 인터페이스 계약 (예)

| 메서드 | 입력 | 출력 | 실패 |
|--------|------|------|------|
| `saveInput(grid, optional metadata)` | 4×4, 메타(타임스탬프 등 선택) | void / 성공 | I/O, 권한 |
| `loadLastInput()` | 없음 | 4×4 또는 empty | 없음 |
| `saveResult(sixTuple)` (선택) | `int[6]` | void | I/O |
| `loadLastResult()` (선택) | — | `int[6]` | — |

**저장 대상:** 입력 행렬(필수), 실행 결과 `int[6]`(선택, 학습용 추천).

### 3.3 구현 옵션 비교

| 옵션 | 내용 | 장점 | 단점 |
|------|------|------|------|
| A InMemory | 프로세스 내 `Map` 또는 단일 슬롯 | 테스트 빠름, 설정 없음 | 재시작 시 소멸 |
| B File (JSON) | `matrix.json` 등 | 재현 가능 | 경로/권한/동시 쓰기 |

**추천: A (InMemory)를 1단계, B (JSON)를 2단계** — TDD·CI에서는 A로 **결정적**·빠른 피드백. “파일” 학습이 목표일 때만 B로 동일 인터페이스 뒤에 교체.

### 3.4 Data 레이어 테스트

| # | 케이스 | 기대 |
|---|--------|------|
| T-01 | save 후 load | grid 동일(4×4) |
| T-02 | load 없이 첫 호출 | 정의된 “empty” 동작(예: empty Optional) |
| T-03 | (File) 손상 JSON | 정의된 예외 + `E_DATA` 매핑은 상위 |
| T-04 | (File) 파일 없음 | empty 또는 기본 |
| T-05 | 잘못된 크기 데이터 로드 시도 | 거부 또는 복구 정책 + INV-01 유지 |

## 4) Integration & Verification (통합 및 검증)

### 4.1 통합 경로

```
[Boundary Input] → [Input validation] → [Application (선택): orchestration]
    → [Domain: analyze + decide] → [Boundary Output]
    ↓
[Data: save/load]  (의존: Application 또는 Boundary의 저장 옵션; Domain은 Data를 참조하지 않음)
```

**의존성 방향:** `Boundary` → `Application`(선택) → `Domain`; `Data`는 **같은 방향**으로만 주입(도메인이 저장소에 의존하지 않음).

### 4.2 통합 테스트 시나리오

**정상 (2 이상)**  
- **IT-01:** 유효 그리드 → 실제 Domain → `int[6]`이 Golden 파일 또는 고정 기대와 일치.  
- **IT-02:** 흐름 끝에서 InMemory `save` → `load` → 동일 입력으로 재실행 → 동일 `int[6]`.

**실패 (3 이상)**  
- **IT-03:** 경계에서 잘못된 크기 → 검증에서 중단, Domain 미호출.  
- **IT-04:** Mock Data가 IOException → `E_DATA`.  
- **IT-05:** (도메인 실패 케이스를 지원하는 경우) 유효 입력이지만 해만 없음 → `E_DOMAIN`.  
- **IT-06:** 중복 값 입력 → `E_DUPLICATE`, Domain 미호출.

### 4.3 회귀 보호 규칙

| 규칙 | 내용 |
|------|------|
| R-01 | `int[6]` 요소 의미·순서·1-index **변경 시** 버전을 올리고 Golden·문서·Traceability를 함께 갱신 |
| R-02 | Public Error `code`·`message` 테이블 변경 시 UI 계약 테스트 전부 재실행 |
| R-03 | Domain invariant ID 추가/삭제 시 대응 단위 테스트 필수 |
| R-04 | 기존 테스트 **삭제 금지**; 동작 변경 시 테스트를 **이전 기대 → 새 기대**로 갱신하고 커밋 메시지에 근거 기재 |

### 4.4 커버리지 목표

| 레이어 | 목표 |
|--------|------|
| Domain | 95%+ |
| UI Boundary | 85%+ |
| Data | 80%+ |
