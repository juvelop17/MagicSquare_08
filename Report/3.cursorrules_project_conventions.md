# MagicSquare — `.cursorrules` 프로젝트 규칙 (정리)

> **원문(SoT):** 저장소 루트의 [`.cursorrules`](../.cursorrules) (YAML)  
> 이 문서는 동일 내용을 Report에서 읽기·검토하기 쉽게 섹션별로 풀어 쓴 사본이다.

---

## 1. project

| 항목 | 내용 |
|------|------|
| **name** | MagicSquare |
| **stack** | Python, pytest, ECB(Clean Architecture 경계), TDD(Red–Green–Refactor) |
| **references** | [2.4x4_magic_square_dual_track_tdd_design.md](2.4x4_magic_square_dual_track_tdd_design.md) |

**설명(요지):** 4×4 마방진 도메인. 부분이 채워진 `int[4][4]` (0=빈칸 2개, 0 또는 1~16 비중복)를 입력·판정해 `int[6] = [r1,c1,n1,r2,c2,n2]`(1-index)로 반환. Magic Constant **34** 및 도메인 INV는 위 설계 문서를 따른다.

## 3. architecture (ECB 3레이어)

| 레이어 | 역할 | import·의존 |
|--------|------|-------------|
| **boundary** | UI, HTTP API, CLI, 파일 I/O 등 **외부 입출력** | `control`만 직접 호출(오케스트레이션). `entity`는 `control`을 통해서만. |
| **control** | 유스케이스·앱 서비스, 흐름·트랜잭션 경계(필요 시) | `entity` |
| **entity** | 도메인 엔터티·VO·도메인 서비스(순수 규칙) | 표준 라이브러리·엔터티 내부 타입만(다른 프로젝트 레이어 의존 금지) |

**의존성 방향(한 방향):** `boundary` → `control` → `entity`  
- `entity`는 `control`·`boundary`에 의존하지 않는다.  
- `control`은 `boundary`에 의존하지 않는다(UI/API 세부에 유스케이스가 묶이지 않게).

**Dual-Track:** 화면 전용 규칙은 `boundary`, 순수 도메인 규칙은 `entity` + `control`에 둔다.

## 4. tdd_rules

### 4.1 red_phase

- **description:** 먼저 **실패하는** 테스트를 작성하고, RED를 관측한 뒤에만 구현으로 넘어간다.  
- **rules:** 구현 전 실패 테스트 추가 · AAA · 의도한 이유로 실패 확인 후 다음 단계 · Report·계약과 정합  
- **must_not:** RED 없이 프로덕션 대량 작성/커밋 · 구현 뒤에 맞추기만 하는 테스트(규격 역전)

### 4.2 green_phase

- **description:** **실패 중인** 테스트를 **최소** 구현으로 통과시킨다.  
- **rules:** 기존 실패 해소 범위만 · GREEN 확인 · 상수→일반화는 TDD로 단계적  
- **must_not:** 이 단계에서 **리팩터링** · 테스트 약화·삭제·부적절한 `skip`/모킹 남용(합의 없이)

### 4.3 refactor_phase

- **description:** **외부 관측 동작**은 유지한 채 구조만 개선한다.  
- **rules:** 중복·이름·레이어 경계 정리 · public API·계약은 보수적으로 유지 · 리팩터 전후 `pytest`로 동일 Green · 커버리지 유지(합의 시 dead code 제거 예외)  
- **must_not:** 이 단계에서 **기능 추가**·새 실패 케이스(새 동작은 Red로) · 테스트 끄기·ECB 깨기

## 5. testing

| 항목 | 값 |
|------|-----|
| **framework** | pytest |
| **pattern** | AAA (Arrange–Act–Assert) |
| **coverage_minimum** | 80% |
| **coverage_scope** | `entity`·`control` 핵심 분기·불변조건 우선, `boundary`는 I/O·통합에서 보완 |
| **naming** | 파일 `test_*.py`, 함수 `test_*` |
| **discovery** | `tests/` |

**fixture_scope(요지)**

| scope | 용도 |
|-------|------|
| **function** | 기본. 테스트 간 격리·재현성 |
| **class** | 동일 `TestClass`에서만 가벼운 공유·SUT 팩토리 |
| **module** | 무거운 **읽기 전용** 자원(대용량 seed 등) |
| **session** | 전역 부작용·느린 외부 자원, 팀이 명시한 경우에만(기본 미사용) |

## 6. forbidden

| pattern | reason | alternative |
|---------|--------|-------------|
| `print()`를 프로덕션/도메인 경로에 | stdout 의존·테스트/운영 불안정·도메인 I/O 결합 | `logging` 또는 boundary의 UI 메시지 |
| 도메인·판정 곳에 **매직** 상수/문자열 산개 | 1-index·34·INV와 계약이 흩어짐 | 이름 있는 상수·Enum·주입(테스트는 픽스처) |
| `except:` / `except Exception:` 만 두고 **삼킴** | 원인·도메인 예외 은닉 | 구체 예외·Result/도메인 예외, boundary에서 메시지화; 광범위 시 로그+재 raise |

## 7. file_structure (ECB 기준, 트리)

원문은 [`.cursorrules`](../.cursorrules)의 `file_structure` 블록과 동일. 요약 트리:

```text
MagicSquare/
├── boundary/           # UI, CLI, API
│   ├── __init__.py
│   ├── cli/
│   └── web/
├── control/            # 유스케이스, 오케스트레이션
│   ├── __init__.py
│   └── use_cases/
├── entity/             # 도메인
│   ├── __init__.py
│   └── ...
├── tests/              # tests/entity, control, boundary 미러
│   ├── __init__.py
│   ├── conftest.py
│   └── ...
├── Report/
├── Prompting/
├── pyproject.toml
└── .cursorrules
```

**원칙:** 프로덕션 소스 루트는 `boundary` / `control` / `entity`만 두고, 테스트는 `tests/`에만 둔다.

## 8. ai_behavior (Cursor AI)

| 시점 | 할 일 |
|------|--------|
| **before_coding** | 관련 `tests/**`·기존 테스트·Report 입출력 계약을 먼저 읽는다. Red에 맞는 **실패 테스트** 위치를 정한 뒤(가능하면 테스트 파일 먼저) 프로덕션을 연다. |
| **during_coding** | ECB `dependency_rule` 준수. 새·수정 심볼에 타입·public docstring·88열. Green에서 리팩터/범위 밖 섞지 않기. `tdd_rules` 위반이 있으면 응답 **맨앞**에 `⚠ TDD rules:` 한 줄 경고. |
| **after_coding** | `pytest` 실행 또는 명령 안내, 실패 시 Red/Green/Refactor로 되돌림 명시. 커버리지 80%·AAA·`forbidden` 잔 점검. |

**한 줄 요약:** 테스트 우선(RED 확인), ECB 경계, 타입+문서, TDD·금지 위반 시 경고 한 줄.
