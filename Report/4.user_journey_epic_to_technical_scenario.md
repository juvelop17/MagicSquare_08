# MagicSquare — 사용자 여정 및 시나리오 명세 (Epic → Level 5)

> Invariant 기반 사고 훈련을 위한 **Epic → Journey → User Story → Technical(Gherkin) → 검증** 산출물 정리  
> 작성 기준일: 2026-04-28

---

## 1. Level 1: Epic (비즈니스 목표)

### Epic 제목

**Invariant(불변조건) 기반 사고 훈련 시스템 구축**

### 목적

4×4 Magic Square 문제를 도메인으로 삼아 다음 역량을 체계적으로 훈련한다.

| 영역 | 설명 |
|------|------|
| **Invariant 중심 설계** | 문제의 핵심 불변조건을 먼저 식별하고, 그에 맞춰 도메인 모델과 동작을 설계하는 사고 방식을 익힌다. |
| **Dual-Track TDD** | 테스트 설계와 구현을 분리해 추적 가능한 방식으로 진행하는 TDD 실무 패턴을 적용한다. |
| **입력·출력 계약 명확화** | 경계값, 오류 케이스, 허용 입력 범위를 계약으로 명시하고 검증하는 습관을 만든다. |
| **설계 → 테스트 → 구현 → 리팩토링** | 한 사이클을 반복 가능한 워크플로로 체화하여, 변경에 안전한 코드 구조를 만든다. |

### 성공 기준

| 기준 | 목표 |
|------|------|
| **Domain Logic 테스트 커버리지** | **95% 이상** |
| **입력 검증** | **계약 테스트 100% 통과** |
| **하드코딩** | **없음** |
| **매직 넘버** | **없음** |
| **추적성** | **Invariant → Test 연결 가능** |

### 한 줄 요약

4×4 Magic Square를 통해 **불변조건·계약·TDD·리팩토링**을 하나의 학습 루프로 묶고, **측정 가능한 품질 기준**으로 완료 여부를 판단한다.

## 2. Level 2: User Journey (사용자 여정)

### Persona

| 항목 | 내용 |
|------|------|
| **역할** | 소프트웨어 개발 학습자 |
| **맥락** | TDD를 반복적으로 연습 중이며, 계층·의존성 방향을 구분하는 **Clean Architecture**를 함께 익히는 단계 |

### Journey

#### Step 1 — 문제 인식

목표는 **“마방진 앱 하나를 만든다”**가 아니라, **불변조건과 계약을 기준으로 한 도메인 문제**로 정의하는 것이다.

- 한 번에 “전체 솔버”가 아니라, **무엇이 가능하고 무엇이 금지되는지**부터 말할 수 있게 한다.
- 구현 범위를 **Blank 찾기 → 빈칸 숫자 결정 → 유효성 검증 → (필요 시) 풀이**처럼 **역할 단위**로 나눌 수 있게 인식을 맞춘다.

#### Step 2 — 계약 정의

구현에 들어가기 전에 **입력·출력·실패**를 문서·테스트 가능한 형태로 고정한다.

| 구분 | 내용 |
|------|------|
| **입력 스키마** | 그리드 크기, 허용 값 범위, 빈칸 표현 방식, 입력 형식 |
| **출력 스키마** | 완성 그리드, 부분 결과, 성공/실패 구분 방식 |
| **예외 정책** | 잘못된 형식, 범위 밖 값, 해 없음 등에 대한 예외 타입·메시지·대체 결과(옵션) |

#### Step 3 — Domain 분리

UI·프레임워크와 무관한 **도메인 단위**로 책임을 나눈다.

| 구성 요소 | 역할(개념) |
|-----------|------------|
| **BlankFinder** | 빈칸 위치·개수 탐색 |
| **MissingNumberFinder** | 주어진 제약 하에서 빈칸에 들어갈 수 있는 후보/값 결정 |
| **MagicSquareValidator** | 완성(또는 부분) 그리드가 마방진 조건을 만족하는지 검증 |
| **Solver** | 위 구성요소를 조합해 **입력 계약을 만족하는 출력**을 만들거나, 계약에 맞게 실패를 반환 |

#### Step 4 — Dual-Track 진행

| 단계 | 내용 |
|------|------|
| **RED (병렬)** | **UI RED** / **Logic RED** |
| **GREEN** | 각 트랙에서 **최소한으로 통과**시키는 구현만 추가 |
| **REFACTOR** | 중복 제거, 경계 정리, 의존성 방향 점검 후 **두 트랙을 하나의 설계로 정리** |

#### Step 5 — 회귀 보호

| 유형 | 예시 |
|------|------|
| **엣지 케이스** | 경계 값, 단일·다중 빈칸, 대칭·동일 합 등 |
| **입력 오류 케이스** | 스키마 위반, 빈 입력, 타입·길이 불일치 |
| **조합 실패 케이스** | 모순·무해·부분 완성 후 막힘 등 실패 경로 |

### Journey 한 줄 요약

**계약으로 문제를 고정하고 → 도메인을 쪼개고 → Dual-Track으로 검증·구현한 뒤 → 회귀 테스트로 지속 가능하게 만든다.**

## 3. Level 3: User Stories

### Story 1 — 입력 검증

**As a** 학습자  
**I want to** 입력이 **정확히 4×4**이고, 문제 도메인 규칙을 만족하는지 검증되길 원한다  
**So that** 잘못된 데이터가 Domain 계층으로 넘어가지 않도록 한다  

**Acceptance Criteria**

- 행·열 개수가 4×4가 아니면 예외  
- 빈칸(`0`)이 **정확히 2개**가 아니면 예외  
- 1~16 범위 밖 값이 있으면 예외  
- 1~16 중 **중복**이 있으면 예외  

---

### Story 2 — 빈칸 탐색

**As a** 학습자  
**I want to** 값이 `0`인 칸의 **좌표를 정확히** 알고 싶다  
**So that** 이후 조합·배치 시도가 가능하도록 한다  

**Acceptance Criteria**

- **row-major** 순서로 빈칸을 나열한다  
- 빈칸 좌표는 **정확히 2개** 반환한다  

---

### Story 3 — 누락 숫자 탐색

**As a** 학습자  
**I want to** 1~16 중 그리드에 **나타나지 않은 숫자 두 개**를 알고 싶다  
**So that** 두 빈칸에 넣을 후보 조합을 일관되게 정의할 수 있다  

**Acceptance Criteria**

- 누락된 숫자는 **1~16 중 정확히 2개**이다  
- 반환 순서는 **오름차순**이다  

---

### Story 4 — 마방진 판정

**As a** 학습자  
**I want to** 완성된 4×4 그리드가 **마방진 조건**을 만족하는지 판정받고 싶다  
**So that** 솔버·검증 로직이 동일한 규칙으로 성공/실패를 말할 수 있다  

**Acceptance Criteria**

- **모든 행**의 합이 서로 같다  
- **모든 열**의 합이 서로 같고, 행 합과 동일하다  
- **두 대각선**의 합이 위와 동일하다  

---

### Story 5 — 두 조합 시도

**As a** 학습자  
**I want to** 작은 누락 숫자를 **첫 번째 빈칸**에 먼저 두고 시도하고, 실패하면 순서를 바꿔 **한 번 더** 시도하길 원한다  
**So that** 해 탐색을 고정된 규칙으로 재현 가능하게 한다  

**Acceptance Criteria**

- 첫 시도: **작은 누락 숫자 → 첫 번째 빈칸**(row-major 상 앞쪽 `0`), 큰 누락 숫자 → 두 번째 빈칸  
- 그 배치가 마방진이 아니면: **두 숫자를 빈칸에 대해 reverse**로 두 번째 시도  
- 성공 시 반환하는 **정답 배열의 길이는 6** (예: `[r1,c1,r2,c2,v1,v2]` 등 프로젝트에서 한 번 정의)  

---

### 추적용 요약

| ID | 제목 | 핵심 검증 |
|----|------|-----------|
| US-1 | 입력 검증 | 4×4, 빈칸 2, 무중복, 범위 1~16 |
| US-2 | 빈칸 탐색 | row-major, 좌표 2개 |
| US-3 | 누락 숫자 | 2개, 오름차순 |
| US-4 | 마방진 판정 | 행·열·대각선 합 동일 |
| US-5 | 두 조합 시도 | small→first, 실패 시 reverse, 정답 길이 6 |

## 4. Level 4: 구현 시나리오 — Technical (Gherkin)

```gherkin
Feature: 4x4 마방진 완성

  불변조건 기반 로직을 검증하기 위해
  TDD를 연습하는 개발자로서
  부분적으로 채워진 4x4 마방진을 완성하고 싶다

  Background:
    Given 4x4 행렬이 주어지고
    And 0은 빈칸을 의미하며
    And 정확히 2개의 셀이 0을 포함하고 있고
    And 숫자는 1부터 16 사이여야 하며
    And 0을 제외한 중복 숫자는 허용되지 않으며
    And 4x4의 마방진 상수는 34이다

  Scenario: 작은 수 → 큰 수 순서로 마방진이 완성되는 경우
    Given 다음과 같은 행렬이 주어졌을 때:
      | 16 |  2 |  3 | 13 |
      |  5 | 11 | 10 |  8 |
      |  9 |  7 |  0 | 12 |
      |  4 | 14 | 15 |  0 |
    When 시스템이 빈칸 좌표를 찾고
    And 누락된 두 숫자를 찾은 뒤
    And 작은 숫자를 첫 번째 빈칸에 배치하고
    And 큰 숫자를 두 번째 빈칸에 배치하면
    Then 모든 행의 합은 34여야 하고
    And 모든 열의 합은 34여야 하며
    And 두 대각선의 합도 34여야 하고
    And 결과는 길이 6의 배열로 반환되어야 하며
    And 반환되는 좌표는 1-index 기준이어야 한다

  Scenario: 역순 배치 시 마방진이 완성되는 경우
    Given 다음과 같은 행렬이 주어졌을 때:
      | 16 |  2 |  3 | 13 |
      |  5 | 11 | 10 |  8 |
      |  9 |  7 |  0 | 12 |
      |  4 | 14 | 15 |  0 |
    When 작은 숫자를 첫 번째 빈칸에 배치했을 때 마방진이 되지 않고
    And 큰 숫자를 첫 번째 빈칸에 배치했을 때 마방진이 되면
    Then 시스템은 역순 배치 결과를 반환해야 하며
    And 최종 행렬은 마방진 상수 34를 만족해야 한다

  Scenario: 빈칸 개수가 올바르지 않은 경우
    Given 행렬에 빈칸이 1개만 존재할 때
    When 유효성 검증을 수행하면
    Then 오류가 발생해야 한다

  Scenario: 중복 숫자가 존재하는 경우
    Given 0을 제외한 중복 숫자가 포함된 행렬일 때
    When 유효성 검증을 수행하면
    Then 오류가 발생해야 한다

  Scenario: 값의 범위를 벗어난 경우
    Given 행렬에 16을 초과하는 숫자가 포함되어 있을 때
    When 유효성 검증을 수행하면
    Then 오류가 발생해야 한다
```

### 구현 시 참고

| 항목 | 설명 |
|------|------|
| **길이 6 배열** | 예: `[r1,c1,r2,c2,v1,v2]` — 좌표 두 개(1-index) + 채운 값 두 개. 프로젝트에서 한 번만 문서·테스트로 고정할 것. |
| **Scenario 1 vs 2** | 동일한 `Given` 표에 대해 “작은 수→큰 수 성공”과 “역순만 성공”은 보통 **서로 다른 입력 행렬**로 분리하는 것이 일관적이다. 필요 시 Scenario 1 전용 예제 행렬을 별도로 둔다. |
| **오류 시나리오** | `Examples` 테이블로 구체 행렬을 넣으면 회귀 테스트에 유리하다. |

## 5. Level 5: 시나리오 검증 및 정리

### 시나리오 완성도 체크리스트

#### 4레벨 일관성

**Epic → Journey**

| 검증 항목 | 확인 질문 |
|-----------|-----------|
| Epic의 성공 지표가 Journey에 반영되는가? | 커버리지·계약 테스트·하드코딩 금지·Invariant 추적이 Journey 단계에 대응되는가? |
| Journey의 모든 단계가 Epic 목표에 기여하는가? | 각 Step이 Invariant / Dual-Track TDD / 계약 / 설계→리팩터 루프와 연결되는가? |
| Pain Point가 명확한가? | 계약 없이 구현, 도메인 혼합, 테스트 추적 단절 등이 문서화되어 있는가? |

**Journey → Story**

| 검증 항목 | 확인 질문 |
|-----------|-----------|
| Journey의 각 Stage마다 최소 Story 1개인가? | 입력 검증, 빈칸·누락 숫자, 판정, 두 조합 시도가 Journey와 매핑되는가? |
| Story가 구체적인 기능으로 변환되는가? | 각 Story의 검증 대상 모듈·행위를 한 문장으로 설명 가능한가? |
| Acceptance Criteria가 측정 가능한가? | 통과/실패로 판정 가능하고 테스트와 1:1 대응 가능한가? |

**Story → Technical**

| 검증 항목 | 확인 질문 |
|-----------|-----------|
| 모든 AC가 Gherkin 시나리오로 변환되는가? | US-1~5 핵심 AC가 Feature/Scenario에 반영되는가? |
| Given–When–Then이 명확한가? | 전제·행위·기대가 섞이지 않았는가? |
| 테스트 자동화 가능한가? | 동일 입력에 대해 결정적 출력·예외로 단언 가능한가? |

#### Edge Case 커버리지

**정상**: Happy Path(첫 시도 성공 / 역순 성공) 시나리오 존재  

**예외** (본 도메인 기준): 4×4 아님, 빈칸 ≠ 2, 중복, 범위 밖, 두 조합 모두 실패(정책에 따라 명시)  

**경계**: 1, 16, `0` 표현, row-major 첫/끝 빈칸, null/빈 배열/길이 불일치  

#### 사용자 중심성 (학습자 Persona)

동료·멘토와 Story·AC 리뷰, Journey 검증, 피드백을 AC·Gherkin에 반영. (선택) Journey 단계별 감정 흐름(막힘 → 계약·테스트로 안정) 기술.

#### 구현 가능성

| 영역 | 확인 |
|------|------|
| 기술 | 선택 언어·테스트 프레임워크에서 시나리오 또는 예제 테이블 자동화 가능 여부 |
| 알고리즘·정책 | 빈칸 순서, 누락 숫자, 두 번 시도 규칙이 코드로 구현 가능한지 |
| 데이터 | 그리드 표현, 길이 6 의미, 1-index 좌표가 한곳에 정의되어 있는지 |
| 경계 | CLI/API/UI 중 무엇을 쓰든 입출력 계약 초안 존재 여부 |

### 한 페이지 요약

| 레벨 | 검증 한 줄 |
|------|------------|
| **L1 Epic** | 성공 지표가 숫자·금지 사항·추적성으로 측정 가능한가? |
| **L2 Journey** | 단계마다 계약·도메인·TDD·회귀가 드러나는가? |
| **L3 Story** | AC가 테스트 assertion으로 줄어드는가? |
| **L4 Technical** | Given–When–Then이 자동 테스트 스텝으로 옮겨지는가? |
| **L5 정리** | 위가 모두 충족되면 시나리오 세트는 구현·교육 목적에 완결에 가깝다. |
