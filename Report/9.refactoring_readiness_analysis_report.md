## 4. ECB(Entity-Control-Boundary) 분석

### 4.1 현재 역할 판단

- `magicsquare/domain.py`
  - 현재는 **Entity 성격**이 강함(순수 도메인 연산: 빈칸 좌표 탐색).
- `magicsquare/boundary.py`
  - 입력 검증/오류 변환/외부 API 위임을 수행하는 **Boundary 역할**을 수행 중.
  - 다만 Boundary 내부에서 다른 Boundary API를 감싸는 구조라 경계 중복 가능성이 있음.
- `magicsquare/gui/app.py` *(질의의 `gui/main_window.py`에 대응하는 현재 파일)*
  - 기본은 UI지만, 클릭 핸들러에서 검증/실행 흐름 제어까지 맡아 UI 외 책임이 일부 섞여 있음.

### 4.2 위치 조정 권고

- `magicsquare/boundary.py`의 검증 책임은 `validator` 계층으로 일원화 고려.
- `magicsquare/gui/app.py`의 `_on_solve_clicked()` 내 유스케이스 흐름 조율은 Boundary/Control 서비스로 분리 고려.
- `magicsquare/domain.py`의 순수 도메인 규칙 함수는 Entity 계층으로 명시 정렬(패키지 일관성) 고려.

## 5. SRP(단일 책임 원칙) 위반 포인트

- `magicsquare/domain.py:8-21`
  - `find_blank_coords()`가 좌표 추출(탐색) + 규칙 검증(빈칸 개수 강제)을 동시에 담당.
- `magicsquare/gui/app.py:68-76`
  - `_on_solve_clicked()`가 UI 이벤트 처리 + 검증/실행 흐름 분기 + 실패 정책 반영을 동시에 담당.

## 7. 리팩토링 후 검증 방법

### 7.1 회귀 테스트 명령어

- `pytest -q`
- `pytest -q --cov=magicsquare --cov-report=term-missing`

### 7.2 외부 동작 불변 확인

- 동일 입력에 대해 `validate()/solve()`의 반환/예외 계약이 변경되지 않았는지 비교 테스트 수행
- `python -m magicsquare.gui` 실행 후 정상/오류 경로에서 라벨 및 경고창 동작 동일성 확인
- 도메인 규칙(빈칸 2개, row-major, 1-index) 결과 불변성 확인

## 8. 결론

- 현재 코드는 기능적으로 작동 가능한 MVP이며, 리팩토링은 가능하다.
- 다만 GUI 이벤트 핸들러와 Boundary 경계 중복 지점을 먼저 정리해야 유지보수성이 크게 개선된다.
- 리팩토링 착수 전 `solve()`/GUI 동작 테스트를 보강해 회귀 안전망을 확보하는 것이 필수다.
