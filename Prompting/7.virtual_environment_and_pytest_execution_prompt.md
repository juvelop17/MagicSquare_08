README에는 이미 TDD 사이클(RED→GREEN→REFACTOR)이 잘 정리돼 있고, Git과 맞추는 방법은 **“브랜치 단위”**와 **“커밋 단위”** 두 가지 관점으로 나누면 됩니다.

## 1. 추천: 기능 브랜치 + 커밋으로 RGR 표현

브랜치는 **User Story(또는 Epic 단위)** 하나로 두고, **같은 브랜치 안에서** RED → GREEN → REFACTOR를 **커밋으로 나누는** 방식이 가장 흔합니다.

| 단계 | 하는 일 (예시) |
|------|----------------|
| **RED** | 실패하는 테스트만 추가 → 커밋 (이 시점에 CI는 빨간색이어도 됨; 로컬에서는 `pytest`로 의도된 실패 확인) |
| **GREEN** | 테스트를 통과하는 최소 구현만 → 커밋 |
| **REFACTOR** | 동작 유지한 정리만 → 커밋 (테스트 전부 통과 유지) |

브랜치 이름 예: `feature/US-001-input-validation`, `feature/us-002-empty-cells`처럼 README의 US와 맞추면 추적이 쉽습니다.

커밋 메시지에 단계를 박아 두면 히스토리만 봐도 RGR가 보입니다.

- `test(us-001): add failing cases for ...` (RED)
- `feat(us-001): minimal validation for ...` (GREEN)
- `refactor(us-001): extract helper ...` (REFACTOR)

Conventional Commits 스타일(`test:` / `feat:` / `refactor:`)과도 잘 맞습니다.

## 2. 선택: RED만 따로 브랜치로 빼기

팀 규칙으로 **“메인/통합 브랜치에는 항상 녹색만”**을 강제하면 RED 커밋을 올리기 어렵습니다. 그때는 예를 들어:

- `red/US-001` → 실패 테스트만 넣고 PR (머지 전랭크는 팀 규칙에 따라)
- 또는 로컬/드래프트 PR에서만 RED 유지하고, GREEN까지 한 뒤에만 리모트에 푸시

같이 정하면 됩니다. **솔로·학습 프로젝트**면 보통 **feature 브랜치 하나에 RED 커밋을 포함**해도 되고, CI에서 해당 브랜치 빌드가 빨간색인 걸 허용할지 정하면 됩니다.

## 3. “브랜치 이름을 red/green/refactor로 나눌까?”

가능은 하지만 **`feature/US-xxx` 하나에 커밋으로 RGR를 쌓는 편**이 관리 부담이 적습니다. 브랜치를 RED/GREEN/REFACTOR마다 따로 만들면 머지·리베이스가 많아져서 이 저장소 규모에는 과한 경우가 많습니다.

## 4. README와의 연결

README의 표(RED에서 프로덕션만 먼저 쓰지 않기, GREEN에서 리팩터링 금지 등)는 **브랜치 이름**보다 **커밋 순서와 리뷰 시점**으로 검증하는 게 자연스럽습니다. PR 설명에 “이 PR은 US-001 GREEN까지 포함, REFACTOR는 다음 커밋”처럼 적어 두면 리뷰어와 나 자신이 단계를 지키기 쉽습니다.

원하시면 README의 「TDD 사이클과 리팩터링」 절 아래에 **“Git: feature 브랜치 + RED/GREEN/REFACTOR 커밋 권장”** 한 단락을 문서로 추가하는 초안도 만들어 드리겠습니다.
