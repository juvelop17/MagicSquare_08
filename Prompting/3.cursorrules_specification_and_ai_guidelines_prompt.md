## 검토 결과 (문제점만)

### 1. YAML 문법
- **문제 없음.** 최상위 키·들여쓰기·한글 스칼라 문자열은 일반적인 YAML 파서 기준으로 유효해 보입니다. `#` 로만 된 줄은 전부 주석이라 구조를 깨지 않습니다.

### 2. 누락된 필수 섹션
- **구조:** 요청하신 최상위 섹션(`project` ~ `ai_behavior`)은 빠진 키 없이 모두 있습니다.
- **내용:** `tdd_rules`를 제외한 섹션은 값이 비어 있어, “프로젝트 규칙 문서”로 보면 **대부분의 섹션이 아직 정의되지 않은 상태**입니다. Cursor가 공식으로 요구하는 고정 스키마는 없지만, 목적상 필수로 채우려던 내용이 있다면 그 부분은 **누락**으로 볼 수 있습니다.

### 3. `tdd_rules`와 `forbidden` 충돌
- **`forbidden`이 비어 있어**, 파일 안에 적힌 규칙끼리의 **논리 충돌은 없습니다.**  
- 다만 나중에 `forbidden`에 “테스트 없이 프로덕션 코드 작성 금지” 등을 넣고, 다른 곳에서 “빠른 스파이크만 허용” 같은 문구를 두면 그때 충돌 가능성이 생깁니다. **현재 파일만 놓고 보면 충돌 없음.**

### 4. `ai_behavior` — 따를 수 없는 규칙
- **`ai_behavior`가 비어 있어** 검토할 규칙이 없습니다. 이후 채울 때만 주의하면 됩니다.
- 참고: 에디터가 이 파일을 **YAML로 검증·실행하지 않고** 프롬프트 성격의 텍스트로 넣는 경우가 많아, “항상 CI 통과까지 확인”처럼 **모델이 환경에서 할 수 없는 행위**를 적으면 실제로는 지키기 어렵습니다. 지금 파일에는 해당 문구가 없습니다.

---

**요약:** 문법·섹션 키·규칙 간 충돌은 현재 파일 기준으로 문제 없음. 다만 **`tdd_rules` 외 전부 공백**이라 규칙 세트로는 아직 거의 비어 있다고 보는 것이 맞습니다.

  - style_guide: PEP8 엄격 준수
  - type_hints: 모든 함수 파라미터와 반환값에 필수
  - docstring: Google 스타일, 모든 public 메서드에 필수

architecture:
  ECB 패턴 3 레이어를 각각 정의해줘:
  - boundary: 외부 입출력 담당 (UI, API, CLI)
  - control: 비즈니스 로직 담당
  - entity: 도메인 데이터 및 규칙 담당
  레이어 간 의존성 방향도 명시해줘.

tdd_rules:
  지금은 문자열 1줄인데, 하위 항목으로 세분화해줘.
  각 phase마다:
    - description: 단계 설명
    - rules: 지켜야 할 규칙 목록
    - must_not: 이 단계에서 하면 안 되는 것

testing:
  - framework: pytest
  - pattern: AAA (Arrange-Act-Assert)
  - coverage_minimum: 80%
  - fixture_scope: 규칙 정의
  - naming_convention: test_ 접두사 필수

forbidden:
  항목마다 아래 구조로 작성:
    pattern: 금지 패턴
    reason: 금지 이유
    alternative: 대신 써야 할 것
  최소 포함 항목: print(), 하드코딩 상수, except 단독 사용

file_structure:
  ECB 기준 폴더 구조를 트리 형태 주석으로 작성해줘.
  boundary/, control/, entity/, tests/ 포함.

ai_behavior:
  Cursor AI가 코드 생성 전·중·후에 반드시 따라야 할 규칙.
  최소 포함:
    - 코드 작성 전 관련 테스트 파일 확인
    - ECB 레이어 경계 위반 금지
    - 타입힌트 없는 함수 생성 금지

프로젝트가 마방진·TDD 설계 문서 중심임을 반영해 `.cursorrules` 전체를 작성합니다.



라인 75에서 큰따옴표가 문자열 중간에서 닫혀 YAML 파서가 깨집니다. 전체를 따옴표로 감쌉니다.



`except:`에 포함된 `:`가 YAML에서 매핑으로 해석됩니다. 해당 `pattern` 값을 따옴표로 감쌉니다.



저장 시 YAML 파서가 깨지지 않도록 다음만 조정했습니다.

- `green_phase.must_not` 안의 `"나중에 고칠"`은 전체를 작은따옴표 한 줄로 감쌈.
- `forbidden`의 `except:` 패턴은 값 안의 `:` 때문에 전체를 큰따옴표로 감쌈.

`python -c "import yaml; yaml.safe_load(...)"`로 검증했습니다.

```yaml
################################################################################
project:
  name: MagicSquare
  description: >
    4×4 마방진(Magic Square) 문제 인식·검증·생성 로직을 다루는 Python 프로젝트.
    설계 문서(Prompting/, Report/)와 동기화된 ECB 아키텍처와 TDD(red-green-refactor)를
    코드에 일관되게 적용한다.
  language: Python
  primary_goals:
    - 마방진 규칙에 맞는 도메인 모델과 알고리즘을 entity/control에 둔다.
    - CLI·스크립트·향후 API는 boundary에만 둔다.
    - 테스트 커버리지와 타입 안전성으로 리팩터링 여지를 남긴다.

################################################################################
code_style:
  python_version: "3.10+"
  style_guide: PEP8 엄격 준수
  type_hints: 모든 함수 파라미터와 반환값에 필수
  docstring: Google 스타일, 모든 public 메서드에 필수
  max_line_length: 88

################################################################################
architecture:
  pattern: ECB (Entity-Control-Boundary)
  layers:
    boundary:
      role: 외부 입출력 담당
      contains:
        - UI (프롬프트, 출력 포맷)
        - API (향후 HTTP 등 진입점)
        - CLI (argparse, 콘솔 진입 스크립트)
      rules:
        - 사용자 문자열·HTTP 요청 등 외부 표현만 다룬다.
        - 도메인 규칙은 직접 구현하지 않고 control을 호출한다.
    control:
      role: 비즈니스 로직·유스케이스 오케스트레이션 담당
      contains:
        - 마방진 검증·생성 절차의 흐름 제어
        - boundary와 entity 사이의 조정(coordination)
      rules:
        - entity만 알고 있는 규칙을 조합한다.
        - I/O나 프레임워크 특화 타입에 의존하지 않는다.
    entity:
      role: 도메인 데이터 및 규칙 담당
      contains:
        - 격자(Grid)·셀(Cell)·합계 검증 등 순수 도메인 모델
        - 마방진 불변식(invariant)과 도메인 연산
      rules:
        - 프레임워크·입출력 모듈을 import하지 않는다.
      dependency_direction: "의존성 없음(순수 도메인)"
  dependency_rule: >
    boundary → control → entity 한 방향만 허용한다.
    entity는 control·boundary를 알지 못한다.
    control은 boundary를 알지 못한다(역방향 의존 금지).

################################################################################
tdd_rules:
  red_phase:
    description: 실패하는 테스트를 먼저 추가해 요구사항을 실행 가능한 스펙으로 고정한다.
    rules:
      - 새 동작·버그 수정 시 테스트를 프로덕션 코드보다 먼저 작성한다.
      - 테스트는 예상 실패(RED)를 한 번 확인한 뒤에만 구현으로 넘어간다.
      - 실패 원인이 요구사항과 일치하는지 확인한다(불안정한 테스트 금지).
    must_not:
      - RED 확인 없이 프로덕션 코드를 먼저 작성하지 않는다.
      - 테스트를 통과시키기 위해 스펙을 축소·회피하지 않는다.
  green_phase:
    description: 테스트를 통과하는 가장 단순한 구현만 추가한다.
    rules:
      - 최소한의 코드로 테스트를 GREEN으로 만든다.
      - ECB 레이어에 맞는 위치에만 코드를 둔다.
      - 타입 힌트와 공개 API docstring 요구를 만족시킨다.
    must_not:
      - 이 단계에서 구조 개선·이름 변경 등 리팩터링을 하지 않는다.
      - '"나중에 고칠" 대량의 중복·데드 코드를 의도적으로 남기지 않는다(최소 구현만).'
  refactor_phase:
    description: 동작은 유지한 채 읽기 좋고 중복 없는 구조로 정리한다.
    rules:
      - 기능·공개 동작 변경 없이 구조만 개선한다.
      - 테스트 스위트는 전부 통과한 상태에서만 리팩터링한다.
      - 커버리지가 testing.coverage_minimum 미만으로 떨어지지 않게 유지한다.
    must_not:
      - 리팩터링 커밋에 새 기능이나 버그 수정을 섞지 않는다.
      - 커버리지를 깨뜨리는 삭제·우회를 하지 않는다.

################################################################################
testing:
  framework: pytest
  pattern: AAA
  pattern_detail: Arrange(준비) → Act(실행) → Assert(검증) 순서로 테스트 본문을 구성한다.
  coverage_minimum: "80%"
  fixture_scope:
    function: 기본. 각 테스트가 독립적으로 동작하도록 state를 공유하지 않는다.
    module: 동일 모듈 내 비용 큰 읽기 전용 리소스만 공유한다.
    session: 프로젝트 전역으로 안전한 불변 설정에만 사용한다(남용 금지).
    rule: fixture는 명시적 스코프만 사용하고, 테스트 간 가변 공유 상태를 두지 않는다.
  naming_convention:
    files: test_*.py 또는 *_test.py (프로젝트에서 채택한 하나로 통일)
    functions: test_ 접두사 필수 (예 test_magic_square_sum_matches_constant)

################################################################################
forbidden:
  - pattern: 디버깅 목적의 print() 호출
    reason: 표준 출력 의존은 테스트·로깅 추적이 어렵고 레이어 경계를 흐린다.
    alternative: logging 모듈 또는 테스트에서는 assert·fixture로 검증한다.
  - pattern: 매직 넘버·매직 문자열 등 하드코딩 상수
    reason: 마방진 크기·합계 등은 도메인 상수로 한곳에서 관리해야 변경에 안전하다.
    alternative: entity 또는 전용 constants 모듈의 이름 있는 상수(Enum/dataclass)로 둔다.
  - pattern: "except 단독 사용 (except: 또는 BaseException 포괄 남용)"
    reason: 예상치 못한 오류를 삼키거나 디버깅을 방해한다.
    alternative: 구체적 예외 타입을 잡고, 필요 시 로깅 후 재던지거나 경계에서 변환한다.

################################################################################
file_structure: |
  # MagicSquare (ECB 기준 권장 레이아웃)
  #
  # magic_square/
  # ├── boundary/
  # │   ├── __init__.py
  # │   └── cli.py              # CLI 진입 (argparse 등)
  # ├── control/
  # │   ├── __init__.py
  # │   └── magic_square_service.py  # 유스케이스 조정
  # ├── entity/
  # │   ├── __init__.py
  # │   ├── grid.py             # 격자·도메인 규칙
  # │   └── constants.py        # (선택) 도메인 상수
  # └── tests/
  #     ├── __init__.py
  #     ├── boundary/
  #     ├── control/
  #     └── entity/
  #
  # 루트에는 Prompting/, Report/, 설정 파일(pyproject.toml 등)을 둘 수 있다.

################################################################################
ai_behavior:
  before_coding:
    - 변경 대상과 동일 ECB 레이어의 기존 테스트·구현 파일을 먼저 확인한다.
    - 관련 entity/control 경계와 기존 테스트 이름 규칙(test_ 접두사)을 확인한다.
  during_coding:
    - ECB 레이어 의존성 방향을 위반하지 않는다(boundary만 외부 I/O).
    - 타입 힌트 없는 함수·메서드를 새로 만들지 않는다(public 포함).
    - tdd_rules(red/green/refactor) 단계를 혼합하지 않는다.
  after_coding:
    - 테스트 실행 가능 여부와 커버리지 하한을 문맥상 확인한다(요청 시 명령 제안).
  on_tdd_violation:
    description: TDD·ECB·타입 규칙을 깨는 변경을 하려 할 때
    action: 사용자에게 한국어로 위반 단계와 대안(테스트 먼저, 레이어 이동 등)을 경고 문구로 출력한다.
```
