# Magic Square 4x4 TDD guidelines
_Exported on 4/28/2026 at 16:08:03 GMT+9 from Cursor (3.2.11)_

---

**User**

당신은 Dual-Track UI(Boundary/Screen) + Logic(Domain) TDD 실무 전문가입니다.
프로젝트: Magic Square 4x4 (TDD 연습용)
이번 작업은 “하나의 커밋” 단위로 진행합니다.

============================================================
중요 규칙 (반드시 준수)
============================================================
- 순서를 반드시 지키십시오: RED 선택 → GREEN 최소 구현 → 테스트 실행 → 통과 확인 → 커밋 요약
- 한 번에 많은 테스트/기능을 추가하지 마십시오. (1개 RED 테스트 묶음만)
- REFACTOR는 이번 커밋에서 하지 마십시오. (GREEN까지만)
- 하드코딩 금지 / 매직 넘버 금지
- MATRIX_SIZE 상수 사용 (Domain·Boundary·GUI에서 격자 차원은 동일 상수 또는 단일 모듈 re-export로 일치)
- Domain(magicsquare/domain.py)은 UI/DB/Web/PyQt **의존 금지**
- GUI는 PyQt 기반 **Screen 레이어**로만 구현하며, 비즈니스 규칙은 Domain에, 입출력·유효성·위임은 Boundary(magicsquare/boundary.py)에 둔다.
- Dual-Track 병렬로 진행:
  UI GREEN = Boundary 계약 + (필요 시) Screen에서의 검증 호출·에러 표시
  Logic GREEN = Domain 규칙

============================================================
고정 계약 (변경 금지)
============================================================
입력:
- 4x4 int[][]
- 0은 빈칸
- 0은 정확히 2개
- 값 범위: 0 또는 1~16
- 0 제외 중복 금지
- 첫 번째 빈칸: row-major 스캔에서 먼저 발견되는 0

출력:
- int[6]
- 좌표는 1-index
- [r1,c1,n1,r2,c2,n2]

============================================================
GUI / 실행 진입점 (이번 작업 범위에 포함)
============================================================
- 의존성: PyQt6 (또는 프로젝트 표준으로 PyQt6 고정) — pyproject.toml에 optional dependency 또는 필수 dependency로 명시
- 진입점: `python -m magicsquare` 또는 `python -m magicsquare.gui` 등 **단일 공식 경로** 1개만 문서화
- magicsquare 패키지에 **Qt를 import 하는 모듈**은 Screen 전용 하위 패키지에 둔다 (예: `magicsquare/gui/` 또는 `magicsquare/screen/`)
  - `domain.py`는 PyQt를 절대 import 하지 않는다.
- 최소 GUI 요구사항 (MVP):
  - 4×4 숫자 입력: QLineEdit 또는 QSpinBox 그리드 등, 빈칸은 0으로 표현
  - 「풀기」버튼: Boundary의 `validate` → `solve` 순으로 호출 (실패 시 ValueError 메시지를 QMessageBox 또는 상태 라벨에 표시)
  - 성공 시 길이 6 결과를 읽기 쉽게 표시 (예: 라벨 또는 읽기 전용 필드)
- GUI 동작에 대한 **자동화 테스트는 이번 커밋에서 필수 아님** (수동 체크리스트만 커밋 요약에 1~3줄). 단, 기존 pytest RED/GREEN 경로는 깨지 않게 유지.

============================================================
Step 1 — 이번 커밋에서 선택할 “첫 RED 테스트”
============================================================
(예시 — 실제 커밋마다 1묶음만 선택)

[TRACK B: Logic]
- L-RED-01: find_blank_coords()가 row-major 순서로 빈칸 좌표 2개를 반환한다.

[TRACK A: UI/Boundary]
- U-RED-01: 입력이 4x4가 아니면 예외를 발생시킨다.
- U-RED-02: 빈칸이 2개가 아니면 예외를 발생시킨다.

👉 (특정 커밋 예시) Logic: L-RED-01, UI: U-RED-01

선택 이유를 2줄로 설명하십시오.

============================================================
Step 2 — GREEN 최소 구현 (Dual-Track 병렬)
============================================================
[TRACK B — Logic GREEN]
- 선택한 Logic RED 1묶음만 통과하는 Domain 최소 구현.

[TRACK A — UI/Boundary GREEN]
- 선택한 Boundary RED 1묶음만 통과하는 validate 등 최소 구현.
- GUI 진입점 커밋은 **별도 커밋**으로 할 것: “실행 가능한 PyQt 윈도우 + pyproject 의존성 + README 실행 한 줄”만, Domain/Business 범위 확장 금지.

============================================================
Step 3 — 테스트 실행 및 통과 확인
============================================================
- 해당 커밋에서 선택한 pytest 묶음만 실행
- GUI는 수동으로 앱 실행 1회 확인 (통과/실패 요약에 한 줄)

============================================================
출력 형식 (반드시 준수)
============================================================
1) 이번 커밋 목표 (1~2문장)
2) 선택한 RED 테스트 (ID + 설명)
3) GREEN 최소 구현 범위 (Track A / Track B / GUI 여부)
4) 실행한 테스트 목록
5) 테스트 결과 요약 (PASS/FAIL)
6) 커밋 메시지 제안 (Conventional Commit)
7) 변경 파일 목록
8) 다음 커밋에서 진행할 RED 또는 GUI 보강 후보 2개

============================================================
주의
============================================================
- REFACTOR 금지
- 한 커밋에 “모든 RED 해결 + 풀 GUI”를 넣지 말 것
- PyQt는 Screen에만; Clean Architecture 경계 유지

**Cursor**

요청하신 규칙에 맞춰 이번 턴은 **한 커밋 분량**만 진행하겠습니다. 먼저 현재 테스트/코드 구조를 빠르게 확인한 뒤, Logic 1묶음 + UI/Boundary 1묶음의 RED를 선정하고 바로 GREEN 최소 구현까지 적용하겠습니다.

현재 저장소는 `magic_square` 구조라서, 요청하신 `magicsquare`(domain/boundary/screen) 계약과 이름이 다릅니다. 기존 테스트를 깨지 않으면서 이번 커밋 범위(RED 1묶음+GREEN 최소)로 맞추기 위해 관련 파일을 읽고 최소 추가 방식으로 진행하겠습니다.

선택은 예시와 동일하게 **L-RED-01 + U-RED-01**로 가겠습니다. 먼저 테스트 두 개를 추가해 RED를 확인하고, 그 다음에 `magicsquare` 패키지에 최소 구현만 넣겠습니다.

RED 실패를 확인했습니다(`magicsquare` 모듈 미존재). 이제 GREEN 단계로 `magicsquare`의 최소 파일만 추가해서, 선택한 두 테스트만 통과시키겠습니다.

1) **이번 커밋 목표 (1~2문장)**  
`Dual-Track` 기준으로 이번 커밋은 **Logic 1묶음 + UI/Boundary 1묶음**의 RED를 먼저 고정하고, 해당 테스트만 통과하는 최소 GREEN 구현만 추가했습니다.  
REFACTOR/GUI 확장은 하지 않고, `MATRIX_SIZE` 상수를 사용하는 최소 골격만 만들었습니다.

2) **선택한 RED 테스트 (ID + 설명)**  
- **L-RED-01**: `find_blank_coords()`가 row-major 순서로 빈칸 좌표 2개를 반환  
- **U-RED-01**: 입력이 4x4가 아니면 Boundary `validate()`가 예외 발생  

선택 이유(2줄):  
- 두 테스트는 Domain 규칙(row-major)과 Boundary 계약(크기 검증)을 동시에 고정해 Dual-Track 시작점으로 적합합니다.  
- 구현 범위가 작아 한 커밋에서 GREEN 최소 구현까지 안정적으로 닫을 수 있습니다.

3) **GREEN 최소 구현 범위 (Track A / Track B / GUI 여부)**  
- **Track A (UI/Boundary)**: `magicsquare/boundary.py`에 `validate()` 추가, 4x4가 아니면 `ValueError` 발생  
- **Track B (Logic/Domain)**: `magicsquare/domain.py`에 `find_blank_coords()` 추가, row-major + 1-index 좌표 반환  
- **공통 상수**: `magicsquare/constants.py`에 `MATRIX_SIZE`, `EMPTY_VALUE` 정의  
- **GUI 여부**: 이번 커밋은 **GUI 미포함** (요구사항에 따라 GUI 진입점은 별도 커밋으로 분리)

4) **실행한 테스트 목록**  
- RED 확인:  
  - `python -m pytest tests/test_magicsquare_domain_blank_coords.py tests/test_magicsquare_boundary_validate_size.py -q`  
- GREEN 확인(동일 대상 재실행):  
  - `python -m pytest tests/test_magicsquare_domain_blank_coords.py tests/test_magicsquare_boundary_validate_size.py -q`  
- GUI 수동 실행:  
  - 이번 커밋 범위 외(미실행)

5) **테스트 결과 요약 (PASS/FAIL)**  
- RED 단계: **FAIL** (의도된 실패, `ModuleNotFoundError: magicsquare`)  
- GREEN 단계: **PASS** (선택한 2개 테스트 통과)  
- GUI 수동 확인: **N/A** (별도 GUI 커밋 예정)

6) **커밋 메시지 제안 (Conventional Commit)**  
- `test+feat(magicsquare): add row-major blank coords and 4x4 boundary validation skeleton`

7) **변경 파일 목록**  
- `magicsquare/__init__.py`  
- `magicsquare/constants.py`  
- `magicsquare/domain.py`  
- `magicsquare/boundary.py`  
- `tests/test_magicsquare_domain_blank_coords.py`  
- `tests/test_magicsquare_boundary_validate_size.py`

8) **다음 커밋에서 진행할 RED 또는 GUI 보강 후보 2개**  
- **후보 A (Boundary RED)**: `U-RED-02` — 빈칸(0) 개수가 정확히 2개가 아니면 예외 발생  
- **후보 B (GUI MVP RED/GREEN)**: PyQt6 Screen 진입점 1개(`python -m magicsquare.gui`) + 4x4 입력/`풀기` 버튼/에러 표시 최소 동작 추가 (Domain 규칙 확장 없이 Boundary 호출만)

**Cursor**

요청하신 대로 이번에는 **GUI 실행 가능한 MVP**를 만들겠습니다. PyQt6 의존성, 단일 진입점, Screen 분리 구조를 추가하고 `validate -> solve` 호출 흐름까지 최소 구현으로 연결하겠습니다.

이제 파일 편집으로 들어가겠습니다. `magicsquare/gui/` 하위에 PyQt Screen을 만들고, `pyproject.toml`에는 PyQt6 의존성/패키지 포함 설정, `README.md`에는 공식 실행 경로 한 줄만 추가하겠습니다.

핵심 GUI 코드는 반영 완료했습니다. 지금부터 기존 pytest 경로가 깨지지 않았는지 확인하고, 마지막으로 GUI 실행 스모크 체크(모듈 진입 확인)까지 하겠습니다.

GUI 모듈 import 실패 원인이 `PyQt6` 미설치로 확인됐습니다. 지금 `.[gui]` 의존성을 설치한 뒤, GUI 진입점까지 다시 스모크 체크하겠습니다.

GUI MVP를 바로 실행 가능한 형태로 추가해뒀습니다. 핵심은 **Screen(PyQt6) 분리 + Boundary 위임 + 단일 실행 경로**입니다.

- `magicsquare/gui/` 추가 (Qt import는 여기에서만)
  - `magicsquare/gui/app.py`: 4x4 `QSpinBox` 입력, `풀기` 버튼, 결과 라벨, 에러 `QMessageBox`
  - `magicsquare/gui/__main__.py`: 공식 진입점
- `magicsquare/boundary.py` 확장
  - `validate(grid)` 후 `solve(grid)` 호출 가능
  - `solve()`는 기존 도메인 플로우를 위임 호출하고 실패 시 `ValueError`로 변환
- `pyproject.toml`
  - `gui = ["PyQt6>=6.7"]` optional dependency 추가
  - 패키지 포함 대상에 `magicsquare*` 추가
- `README.md`
  - 공식 실행 경로 1줄 추가: `python -m magicsquare.gui`

실행 방법:

