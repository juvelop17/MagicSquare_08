## 검토 결과 (문제점만)

### 1. YAML 문법
- **문제 없음.** `yaml.safe_load`로 파싱해 오류가 없었고, 최상위 키는 모두 `null`로, `tdd_rules`만 하위 매핑으로 정상 파싱됨.

### 2. 누락된 필수 섹션
- **키 기준:** 이전에 정한 8개 최상위 키(`project`, `code_style`, `architecture`, `tdd_rules`, `testing`, `forbidden`, `file_structure`, `ai_behavior`)는 **모두 존재**함.
- **내용 기준:** `tdd_rules`만 채워져 있고 나머지 7개는 값이 비어 있음. TDD·품질을 규칙으로 묶을 때 통상 기대하는 **테스트 러너/범위(`testing`)), 금지 사항(`forbidden`)), 에이전트 행동(`ai_behavior`))** 등이 **미정의**라, “규칙 문서”로 쓰기엔 **대부분의 섹션이 비어 있는 상태**로 볼 수 있음. (이것이 “누락”에 해당하는지는 팀이 정한 ‘필수’ 정의에 따름.)

### 3. `tdd_rules`와 `forbidden` 충돌
- **충돌 없음.** `forbidden`이 비어 있어, `tdd_rules`의 green/refactor 문구와 맞닿는 **명시적 금지 규칙이 없음.**

### 4. `ai_behavior` — 따라갈 수 없는 규칙
- **해당 사항 없음.** `ai_behavior`가 **비어 있어** 검토할 실행 불가·모호·상충 규칙이 없음.

---

**요약:** 문법·`tdd_rules`↔`forbidden` 충돌·`ai_behavior` 비현실성 측면에서는 이슈가 없고, **실질적으로는 `tdd_rules`만 있고 나머지 섹션이 비어 있어** 규칙 세트가 얇다는 점이 주된 갭으로 보이면 됨.

MagicSquare 프로젝트 기준으로 작성해.

각 섹션 작성 규칙:

code_style:
  - python_version: "3.10+"
  - style_guide: PEP8 엄격 준수
  - type_hints: 모든 함수 파라미터와 반환값에 필수
  - docstring: Google 스타일, 모든 public 메서드에 필수
  - max_line_length: 88 (Black 기준)

architecture:
  ECB 패턴 3 레이어를 각각 정의해줘:
  - boundary: 외부 입출력 담당 (UI, API, CLI)
  - control: 비즈니스 로직 담당
  - entity: 도메인 데이터 및 규칙 담당
  레이어 간 의존성 방향도 명시해줘.

tdd_rules:
  지금은 문자열 1줄인데, 하위 항목으로 세분화해줘.
  각 phase마다:
    - description: 단계 설명
    - rules: 지켜야 할 규칙 목록
    - must_not: 이 단계에서 하면 안 되는 것

testing:
  - framework: pytest
  - pattern: AAA (Arrange-Act-Assert)
  - coverage_minimum: 80%
  - fixture_scope: 규칙 정의
  - naming_convention: test_ 접두사 필수

forbidden:
  항목마다 아래 구조로 작성:
    pattern: 금지 패턴
    reason: 금지 이유
    alternative: 대신 써야 할 것
  최소 포함 항목: print(), 하드코딩 상수, except 단독 사용

file_structure:
  ECB 기준 폴더 구조를 트리 형태 주석으로 작성해줘.
  boundary/, control/, entity/, tests/ 포함.

ai_behavior:
  Cursor AI가 코드 생성 전·중·후에 반드시 따라야 할 규칙.
  최소 포함:
    - 코드 작성 전 관련 테스트 파일 확인
    - ECB 레이어 경계 위반 금지
    - 타입힌트 없는 함수 생성 금지
    - tdd_rules 위반 시 경고 출력

전체 `.cursorrules` 파일 작성 중입니다.



파일이 잘렸고 `forbidden` 마지막 항목에 `reason`이 중복되었습니다. 수정하고 나머지 섹션을 추가합니다.



중복 `reason` 키 제거 및 `file_structure`, `ai_behavior` 섹션 추가.



작업 반영했습니다. `forbidden` 항목에 있던 중복 `reason` 키는 YAML 오류라 제거했고, `file_structure`·`ai_behavior`를 붙였습니다. 아래가 저장된 `.cursorrules` 전체입니다.

```yaml
################################################################################
project:
  name: "MagicSquare"
  description: |
    4×4 마방진(Magic Square) 도메인을 다루는 Python 프로젝트이다.
    부분이 채워진 격자(int[4][4], 0=빈칸 2개, 0 또는 1~16 비중복)를 입력받아
    빈칸 좌표·누락 수·배치 순서를 판정하고, 결과를 int[6] = [r1,c1,n1,r2,c2,n2] 형식(1-index)으로 반환한다.
    Magic Constant 34 및 도메인 불변조건(INV)은 Report의 도메인 설계를 따른다.
  stack: "Python, pytest, ECB(Clean Architecture 경계), TDD(Red–Green–Refactor)"
  references:
    - "Report/2.4x4_magic_square_dual_track_tdd_design.md"
################################################################################
code_style:
  python_version: "3.10+"
  style_guide: "PEP 8 엄격 준수 (일관된 공백·이름·임포트 순서)"
  type_hints: "모든 함수·메서드의 파라미터와 반환값에 타입 힌트 필수 (typing, Protocol 등 활용)"
  docstring: "Google 스타일 docstring, public 메서드·모듈 export 함수에 필수"
  max_line_length: 88
  tooling_note: "포매터는 Black(88열)을 기본으로 하되, 팀이 확정한 도구가 있으면 그에 맞출 것"
################################################################################
architecture:
  pattern: "ECB (Entity–Control–Boundary) 3레이어"
  layers:
    boundary:
      role: "외부와의 입출력 경계. UI, HTTP API, CLI, 파일 I/O, 사용자/호스트에 직접 노출되는 어댑터"
      may_import: "control만 호출(오케스트레이션). entity는 control을 통해 간접 사용"
    control:
      role: "유스케이스·애플리케이션 서비스. 흐름 제어, 트랜잭션 경계(필요 시), boundary ↔ entity 사이의 비즈니스 절차"
      may_import: "entity"
    entity:
      role: "도메인 엔터티·값 객체·도메인 서비스(순수 규칙). 마방진 판정, 빈칸·누락 수, 합/불변조건 등"
      may_import: "표준 라이브러리·엔터티 내부용 타입만(다른 프로젝트 레이어 의존 금지)"
  dependency_rule: |
    의존성은 항상 바깥 → 안쪽으로만 흐른다: boundary → control → entity.
    entity는 control·boundary에 의존하지 않는다.
    control는 boundary에 의존하지 않는다(유스케이스가 UI/API 세부에 묶이지 않게).
  dual_track_note: "Dual-Track(로직/화면) 설계를 코드 구조에 반영할 때, 화면만의 규칙은 boundary, 순수 도메인 규칙은 entity+control에 둔다."
################################################################################
tdd_rules:
  red_phase:
    description: "새로운 동작·버그에 대해 **먼저** 실패하는 테스트를 작성하는 단계. ‘실패(RED)’를 관측한 뒤에만 구현으로 넘어간다."
    rules:
      - "구현(프로덕션 코드) 추가 전, 실패하는 단위 테스트를 먼저 추가한다."
      - "테스트는 Arrange–Act–Assert 구조로 의도·기대를 명시한다."
      - "테스트를 실행해 **의도한 이유로** 실패하는지 확인한 후 다음 단계로 진행한다(단순 문법 오류는 먼저 고친다)."
      - "도메인 불변조건·유스케이스를 남기는 테스트로 설계(Report·계약)와 정합을 맞춘다."
    must_not:
      - "RED 확인 없이 프로덕션 로직을 먼저 대량 작성·커밋한다."
      - "‘나중에 테스트’로 미루거나, 이미 구현된 동작에 맞춰 테스트만 뒤집는다(테스트가 규격이 되지 않게)."
  green_phase:
    description: "현재 **실패 중인** 테스트를 통과시키는 데 필요한 **최소한의** 구현을 하는 단계."
    rules:
      - "딱 기존 실패를 해소하는 범위에서만 프로덕션 코드를 짠다(과한 일반화·부가 기능 금지)."
      - "테스트를 다시 실행해 **모두 통과(GREEN)**함을 확인한다."
      - "필요 시 가장 단순한 해법(상수 반환→일반화)을 TDD로만 단계적으로 올린다."
    must_not:
      - "이 단계에서 구조·이름·중복 제거를 위한 **리팩터링**을 수행한다(다음 Refactor로 미룬다)."
      - "테스트를 통과시키기 위해 **테스트를 약화·삭제**하거나, 불필요한 `skip`·모킹 남용으로 요건을 비운다(합의되지 않은 경우)."
  refactor_phase:
    description: "동작(테스트가 보장하는 **외부 관측 행동**)은 그대로 두고, 내부 구조만 개선하는 단계."
    rules:
      - "Green 직후, 중복·이름·경계(레이어) 위반이 보이면 리팩터링한다."
      - "엔터티/컨트롤에 어울리는 위치로 코드를 옮기고, public API·계약은 보수적으로 유지·문서 갱신한다."
      - "리팩터링 직전·직후 `pytest` 전체(또는 합의된 하위집합)로 회귀를 돌려 **동일 Green**을 유지한다."
      - "가능한 한 커버리지가 리팩터 전 대비 **떨어지지 않게** 한다(새 dead code 제거는 예외로 합의 가능)."
    must_not:
      - "이 단계에서 기능 추가·새로운 실패 케이스(요구사항 변경)를 섞는다(새 동작이면 먼저 Red로 돌아간다)."
      - "‘대충’ 테스트를 끄거나, ECB 경계를 깨뜨려 편의만 취한다."
################################################################################
testing:
  framework: "pytest"
  pattern: "AAA (Arrange–Act–Assert): 준비·실행·단언을 코드와 빈 줄로 구분해 읽힌다."
  coverage_minimum: "80%"
  coverage_scope: "가능한 한 entity·control에 대해 핵심 분기·불변조건을 커버; boundary는 I/O·통합 범위에서 보완"
  fixture_scope:
    function: "기본. 각 테스트는 서로의 상태에 의존하지 않는다(격리·재현성)."
    class: "동일 TestClass에서만 공유해도 될 **가벼운** 준비·SUT 팩토리."
    module: "모듈 단위로 한 번만 올려도 될 **무거운** 순수 자원(대용량 seed 데이터 로드, 컴파일 캐시 등). side effect 없이 읽기 전용."
    session: "전역 부작용·외부 프로세스가 필요할 때만, 팀이 명시한 경우(기본은 사용하지 않음)."
  naming_convention: "파일 `test_*.py` / 함수 `test_*` 접두사 필수(클래스 `Test*`는 pytest 수집 규칙에 따름)"
  discovery: "`pytest` 기본 discovery 경로 `tests/`를 따른다."
################################################################################
forbidden:
  - pattern: "print() (디버그·로그용 산출을 프로덕션/도메인 경로에 남기기)"
    reason: "표준 출력 의존은 테스트·로그 수집·서버less 환경에서 불안정하며, 도메인 레이어가 I/O에 묶인다."
    alternative: "로깅은 `logging` 모듈(레벨·핸들러 구성) 또는 boundary에서만 사용하는 UI 메시지로 분리한다."
  - pattern: "하드코딩된 ‘마법’ 상수(매직 넘버·문자열)를 도메인·비즈니스 판정 곳곳에 흩뿌리기"
    reason: "의미·단위(1-index, magic constant 34)가 흩어지면 INV·계약 위반과 리팩터 비용이 커진다."
    alternative: "이름 붙은 상수(모듈·클래스 상수)·Enum·설정 주입(테스트는 픽스처/팩토리로 고정 값 제공)."
  - pattern: "except: 또는 except Exception: 만 두고 흡수(로그/재전파 없이 통과)"
    reason: "원인·계약 위반(도메인 예외)이 숨겨지고, 테스트도 실제 실패를 감지하지 못한다."
    alternative: "구체 예외·Result/Either·도메인 예외로 좁히고, boundary에서만 최종 메시지로 변환한다. 정말 광범위일 때는 반드시 로그 후 재 raise 또는 명시적 실패."
################################################################################
file_structure: |
  # MagicSquare/
  # ├── boundary/           # UI, CLI, API 어댑터. 외부 입출력, 사용자 메시지
  # │   ├── __init__.py
  # │   ├── cli/            # (선택) 명령행
  # │   └── web/            # (선택) API
  # ├── control/            # 유스케이스, 앱 서비스, boundary↔entity 오케스트레이션
  # │   ├── __init__.py
  # │   └── use_cases/      # (선택) 유스케이스별 모듈
  # ├── entity/             # 도메인: 그리드, 불변조건, 마방진 판정, 값 객체
```
