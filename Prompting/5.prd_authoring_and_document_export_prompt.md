## 역할별로 보면

1. **`Report/1`** — 제품이 해결하는 문제, 배경, “왜 이 도메인인지” (PRD의 문제 진술·비전에 해당).
2. **`Report/2`** — 요구사항의 뼈대: 규칙·INV·테스트 트랙·완료 조건 (PRD의 기능·비기능·범위).
3. **`Report/4`** — Epic, 페르소나, 여정, Gherkin급 시나리오 (PRD의 사용자·성공 기준·스코프 스토리).
4. **`.cursorrules`** + **`Report/3`** — 구현·프로세스 제약(ECB, red-green-refactor 등). PRD에 “개발 원칙” 또는 부록으로 두기 좋음.

`Prompting/` 아래 파일들은 **그 Report를 만든 프롬프트 템플릿**이라, PRD 문장을 직접 채울 때보다는 **구조를 재사용할 때** 참고하면 됩니다.

## 실무 팁

- PRD를 **처음부터** 쓴다면: **`Report/4`로 목표·여정을 잡고** → **`Report/1·2`로 문제와 규칙을 고정** → **`.cursorrules`/Report 3으로 엔지니어링 원칙을 한 줄 요약**하는 순서가 저장소 설계와 맞습니다.
- **기존 PRD를 갱신**한다면: 변경된 Report만 반영하고, 표의 “소스 정렬” 줄을 최신 파일명에 맞게 고치면 추적이 쉽습니다.

원하시면 `Report/2`·`Report/3` 앞부분만 읽어서 PRD 각 섹션에 어떤 문단을 옮기면 되는지 목차 단위로 짜 드리겠습니다.
