## 1. Executive Summary

본 프로젝트는 4×4 격자에서 **빈칸 2개**와 **누락 숫자 2개**를 고정된 규칙으로 배치해 **행·열·두 주 대각선의 합이 모두 Magic Constant 34**가 되도록 하는 **완성 결과**를, **경계(Boundary)에서의 입력 검증·출력 형식**과 **도메인(Domain)의 순수 판정·조합 시도**로 나누어 구현한다. 알고리즘 난이도 극대화가 아니라 **Dual-Track TDD**(경계 테스트 트랙과 도메인 테스트 트랙), **Invariant 기반 설계**, **입력/출력 계약의 문서·테스트 가능한 고정**을 훈련 목표로 한다. UI·웹·DB에 대한 의존성 없이 **동일 요구사항**을 만족할 수 있도록 하며, 웹 화면이나 영속 저장소는 본 PRD의 완료 범위에 포함하지 않는다.

**훈련하려는 핵심 역량:** 불변조건 식별 및 단위 테스트로의 연결, 입출력 계약(스키마·오류 코드·고정 메시지) 명시, 경계와 도메인의 병렬 TDD 사이클.

## 2. Problem Statement (문제 정의)

### 2.1 정의의 초점

본 제품이 다루는 문제는 “임의의 마방진을 **생성**한다”가 아니라, **부분적으로 채워진** 4×4 격자가 아래 **입력 계약**을 만족할 때, 누락된 두 수를 두 빈칸에 넣어 **모든 행·열·두 주 대각선의 합이 동일(Magic Constant 34)**인 완성 상태에 도달할 수 있는지를 **검증 가능한 절차**(빈칸 탐색 → 누락 수 도출 → 두 번의 고정된 배치 시도 → 판정)로 처리하는 것이다.

### 2.2 입력/출력 계약이 핵심인 이유

- 동일한 자연어 표현(“마방진을 완성한다”)만으로는 빈칸 개수, 좌표 체계, 실패 시 동작이 달라져 **테스트 기대값이 흔들린다**.
- TDD는 **동일 입력에 동일 출력(또는 동일 오류 코드)**을 전제로 하므로, **행렬 표현·0의 의미·빈칸 순서(row-major)·출력 6튜플의 의미**를 계약으로 한 번에 고정해야 한다.
- 경계 계약은 **스키마 위반을 도메인 이전에 차단**하고, 도메인은 **이미 유효한 그리드**에 대해서만 불변조건과 완성 판정을 다룬다.

## 5. Functional Requirements (기능 요구사항)

### 공통 입출력 계약 (고정)

**입력**

- 타입: `int[][]`, **논리적 형태** 4×4 정수 행렬(행 길이 4, 각 행 길이 4).
- `0`: 빈칸.
- 빈칸(`0`) 개수: **정확히 2**.
- 값 허용: 각 셀은 `0` 또는 `1`~`16` 중 하나.
- `0`을 제외한 값은 **중복 불가**.
- **첫 번째 빈칸:** row-major 스캔(행 1→4, 각 행에서 열 1→4) 시 **가장 먼저 만나는** `0`의 칸.

**출력 (성공)**

- 타입: `int[6]`.
- 좌표: **1-index**, 행·열 범위 `1..4`.
- 형식: `[r1, c1, n1, r2, c2, n2]` — `(r1,c1)`은 첫 빈칸, `(r2,c2)`는 둘째 빈칸; `n1`은 첫 빈칸에 배치된 값, `n2`는 둘째 빈칸에 배치된 값.

**누락 숫자 짝**

- `{1..16}`에서 `0`이 아닌 칸에 등장하지 않는 두 정수를 `n_small`, `n_large`라 하되 `n_small ≤ n_large`.

**시도 규칙**

- **시도 1:** 첫 빈칸에 `n_small`, 둘째 빈칸에 `n_large`를 넣은 완전 그리드가 마방진이면, 출력은 시도 1의 배치를 반영한다: `n1 = n_small`, `n2 = n_large`.
- **시도 2:** 시도 1이 마방진이 아니면, 첫 빈칸에 `n_large`, 둘째 빈칸에 `n_small`을 넣어 검사한다. 마방진이면 출력은 **역순 배치**를 반영한다: `n1 = n_large`, `n2 = n_small`.

**시도 1·2 모두 마방진이 아닐 때**

- **실패 정책(고정):** Boundary 또는 Application 오케스트레이션에서 도메인이 “완성 불가”를 반환한 경우, 오류 코드 **`E_DOMAIN`**, 메시지 **`No valid magic square completion exists for this grid.`** (아래 §7·§11과 동일).

### FR-01 입력 검증 (Boundary)

| 필드 | 내용 |
|------|------|
| **Feature ID** | FR-01 |
| **설명** | 호출자가 전달한 4×4 행렬이 구조·INV와 일치하는지 검사한다. 불일치 시 도메인 로직을 호출하지 않는다. |
| **입력** | `int[][]` (위 공통 계약). |
| **처리 규칙** | 행/열 개수 4×4; 각 셀 `0` 또는 1~16; `0` 정확히 2개; 0 제외 값 중복 없음. |
| **출력** | 검증 성공 시 후속 처리로 넘길 수 있는 표현(동일 데이터 또는 검증된 뷰); 실패 시 오류 코드·고정 메시지(§8.1). |
| **승인 기준 (AC)** | (AC-01) 행 수≠4 또는 어떤 행의 길이≠4이면 검증 실패로 분류되고 `E_SIZE`가 기록된다. (AC-02) `0` 개수≠2이면 `E_EMPTY_COUNT`이다. (AC-03) 허용 범위 밖 값이 있으면 `E_VALUE_RANGE`이다. (AC-04) 0 제외 중복이 있으면 `E_DUPLICATE`이다. (AC-05) 위 네 경우 모두에서 도메인의 마방진 판정·해 찾기 함수가 호출되지 않음을 테스트가 단언한다. |
| **오류/예외 정책** | 코드·메시지는 §8.1 표와 문자 단위로 일치해야 한다. |

### FR-02 빈칸 탐색

| 필드 | 내용 |
|------|------|
| **Feature ID** | FR-02 |
| **설명** | 유효한 입력 그리드에서 값이 `0`인 두 칸의 좌표를 row-major 순으로 결정한다. |
| **입력** | INV-01~04를 만족하는 4×4 그리드. |
| **처리 규칙** | 행 우선, 행 번호 오름차순, 동일 행에서는 열 오름차순으로 스캔하여 첫 번째 `0`을 첫 빈칸, 두 번째 `0`을 둘째 빈칸으로 한다. |
| **출력** | 두 좌표 `(r1,c1)`, `(r2,c2)`, 각각 1-index, `1≤r,c≤4`. |
| **승인 기준 (AC)** | (AC-06) 고정된 시드 그리드 A에서 첫 빈칸·둘째 빈칸 좌표가 문서화된 기대와 일치한다. (AC-07) 빈칸이 정확히 2개가 아닌 입력은 FR-01에서 거부되므로 본 기능은 호출되지 않는다. |
| **오류/예외 정책** | 도메인 단독 호출 시 INV-04 위반 입력은 본 PRD 범위에서는 Boundary 선행 검증으로 차단한다; 내부 모듈 테스트에서 중복 정의 시 동일 `E_EMPTY_COUNT` 또는 도메인 전용 실패 결과로 일관되게 처리하고 테스트로 고정한다. |

### FR-03 누락 숫자 탐색

| 필드 | 내용 |
|------|------|
| **Feature ID** | FR-03 |
| **설명** | 1~16 중 그리드에 나타나지 않는 정수 두 개를 찾는다. |
| **입력** | INV-01~04를 만족하는 4×4 그리드. |
| **처리 규칙** | 0이 아닌 칸의 값 집합과 `{1..16}`의 차집합은 항상 크기 2이다. 두 수를 `n_small ≤ n_large`로 명명한다. |
| **출력** | 정수 쌍 `(n_small, n_large)`. |
| **승인 기준 (AC)** | (AC-08) 시드 그리드 A에서 누락 집합이 `{1,6}`으로 단언 가능하다. (AC-09) 검증된 입력에서 누락 개수가 2가 아닌 경우는 발생하지 않는다(INV·FR-01에 의해 배제). |
| **오류/예외 정책** | 논리적으로 불가능한 중간 상태는 범위 밖(테스트로 재현 불가 시 문서화만 하고 구현은 배제). |

### FR-04 마방진 판정

| 필드 | 내용 |
|------|------|
| **Feature ID** | FR-04 |
| **설명** | **0이 없는** 완전히 채워진 4×4 그리드에 대해 마방진 여부를 판정한다. |
| **입력** | 4×4, 모든 셀 1~16, INV-03 만족. |
| **처리 규칙** | 네 행의 합·네 열의 합·주 대각선 합·반대 대각선 합이 **모두 34**이면 참이다. |
| **출력** | 불리언(또는 동등한 명명 가능한 판정 결과). |
| **승인 기준 (AC)** | (AC-10) 알려진 완성 마방진 예제에서 참이다. (AC-11) 한 행의 합만 34가 아닌 완전 그리드(치환으로 구성)에서 거짓이다. |
| **오류/예외 정책** | 완전 그리드가 아니거나 INV 위반인 입력에 대한 호출은 허용되지 않는다; 호출자 테스트에서 금지한다. |

### FR-05 해 찾기(solution): 두 조합 시도 및 반환

| 필드 | 내용 |
|------|------|
| **Feature ID** | FR-05 |
| **설명** | FR-02·FR-03 결과를 사용해 시도 1 후 필요 시 시도 2를 수행하고, 성공 시 `int[6]`을 생성한다. |
| **입력** | 유효 그리드(검증 통과). |
| **처리 규칙** | 시도 1: 첫 빈칸←`n_small`, 둘째 빈칸←`n_large`로 임시 완성 후 FR-04. 참이면 `[r1,c1,n_small,r2,c2,n_large]`. 거짓이면 시도 2: 첫 빈칸←`n_large`, 둘째 빈칸←`n_small` 후 FR-04. 참이면 `[r1,c1,n_large,r2,c2,n_small]`. 둘 다 거짓이면 §5 공통 계약의 실패 정책. |
| **출력** | 성공 시 `int[6]`; 실패 시 `E_DOMAIN` 및 고정 메시지. |
| **승인 기준 (AC)** | (AC-12) 시드 그리드 A에서 시도 1이 성공하면 출력 6원소가 기대 배열과 일치한다. (AC-13) 시도 1 실패·시도 2 성공 전용 시드 그리드 B가 문서화되어 있고 출력이 역순 규칙과 일치한다. (AC-14) 두 시도 모두 실패하는 유효 입력에 대해 `E_DOMAIN`과 고정 메시지가 기록된다. |
| **오류/예외 정책** | `E_DOMAIN`: 입력 스키마는 유효하나 두 배치 모두 마방진이 아님. |

## 6. Business Rules (도메인 규칙)

| ID | 규칙 문장 |
|----|-----------|
| BR-01 | 그리드는 정확히 4행 4열이다. |
| BR-02 | 각 셀의 값은 0 또는 1 이상 16 이하의 정수이다. |
| BR-03 | 값 0인 셀은 정확히 두 개이다. |
| BR-04 | 1~16 범위의 값은 각 값이 최대 한 번만 나타난다(0 제외). |
| BR-05 | 빈칸의 순서는 row-major 순서에서 발견되는 순서이다. |
| BR-06 | 완전 채움에서 모든 행 합·모든 열 합·두 주 대각선 합은 34와 같다(Magic Constant). |
| BR-07 | 출력에서 행·열 좌표는 1-index이며 범위는 1~4이다. |
| BR-08 | 성공 시 `[r1,c1,n1,r2,c2,n2]`에서 `(r1,c1)`은 첫 빈칸, `(r2,c2)`는 둘째 빈칸의 1-index 좌표이고, `n1`은 첫 빈칸에 놓인 값, `n2`는 둘째 빈칸에 놓인 값이다. 시도 1로 성공하면 `n1=n_small`·`n2=n_large`이고, 시도 2로 성공하면 `n1=n_large`·`n2=n_small`이다. |
| BR-09 | 동일 입력에 대한 성공 출력은 항상 동일하다(결정론). |

| NFR-03 | 동일 입력에 대해 성공 경로 출력 바이트/원소 나열이 항상 동일하다 | 결정론 단위·통합 테스트 |
| NFR-04 | 표준 연산으로 호출자가 준 행렬 인스턴스를 **변경하지 않는다**(검증·완성 시도는 복사본 또는 불변 뷰 사용). 테스트: 입력 배열의 참조 동일성 유지 시 원소가 호출 전후 동일함을 단언 | |
| NFR-05 | (선택) 4×4 단일 요청에 대한 처리 시간이 동일 머신에서 **50ms 미만**인 경우가 일반적이다. 성능 테스트는 선택 CI 작업으로만 둔다. | |

## 8. Dual-Track TDD Strategy

### 8.1 Track A — Boundary(UI) TDD

**범위:** 그래픽 UI가 아니라 **계약 경계**(입력 검증, 성공 `int[6]` 전달, 오류 코드·메시지).

**Contract-first 테스트 항목 (예시 ID)**

| ID | 내용 |
|----|------|
| U-01 | 행/열 크기 불일치 → `E_SIZE` |
| U-02 | 빈칸 0개 → `E_EMPTY_COUNT` |
| U-03 | 빈칸 3개 → `E_EMPTY_COUNT` |
| U-04 | 셀 값 17 또는 음수 → `E_VALUE_RANGE` |
| U-05 | 0 제외 중복 → `E_DUPLICATE` |
| U-06 | 도메인 Mock이 고정 `int[6]` 반환 시 경계가 동일 값 전달 |
| U-07 | 도메인이 완성 불가 시 `E_DOMAIN` |

**실패 정책 — 코드 및 메시지 (고정)**

| code | message (정확히 일치) |
|------|------------------------|
| `E_SIZE` | `Grid must be 4 by 4.` |
| `E_EMPTY_COUNT` | `There must be exactly two empty cells (value 0).` |
| `E_VALUE_RANGE` | `Each cell must be 0 or an integer from 1 to 16.` |
| `E_DUPLICATE` | `Values from 1 to 16 must not repeat except for 0.` |
| `E_DOMAIN` | `No valid magic square completion exists for this grid.` |

---

### 8.2 Track B — Domain(Logic) TDD

**메서드/단위 수준 테스트 (예시)**

| ID | 초점 |
|----|------|
| D-01 | 시드 A → `int[6]` 기대 일치 |
| D-02 | 시드에서 빈칸 순서가 row-major와 일치 |
| D-03 | 시도 1 성공 분기 |
| D-04 | 시도 1 실패·시도 2 성공 분기(시드 B) |
| D-05 | 완성 마방진에서 `isMagicSquareComplete` 참 |
| D-06 | 합이 틀린 완전 그리드에서 거짓 |
| D-07 | 경계에서 차단되지만 도메인 단독 시 빈칸 개수 위반 시 실패 경로 |
| D-08 | (경계와 중복 가능) 중복 입력 시 도메인 미호출 |

**불변조건 테스트 매핑**

| Invariant | 테스트에 포함할 단언 |
|-----------|----------------------|
| INV-01 4×4 | 크기 관련 |
| INV-02 값 범위 | 0 및 1~16 |
| INV-03 비중복 | 0 제외 |
| INV-04 빈칸 2개 | 개수 |
| INV-05 합 34 | 행·열·두 대각 |
| INV-06 출력 1-index | 좌표 범위 및 출력 형식 |

---

### 8.3 병렬 진행 규칙

1. **RED (병렬):** Track A에서 계약 실패 케이스용 테스트 추가; Track B에서 도메인 단위 테스트 추가.
2. **GREEN (병렬):** 각 트랙에서 테스트를 통과시키는 **최소** 구현만 추가한다.
3. **REFACTOR:** 중복 제거·이름 정리·의존성 방향 확인 후 전체 스위트 통과를 유지한다.

**금지:** “도메인을 전부 구현한 뒤 경계 검증을 나중에 추가”하는 순서. 경계와 도메인은 **동일 스프린트 내**에서 번갈아 최소 통과를 유지한다.

## 9. Test Plan (QA)

### 9.1 시나리오 기반 목록

| 시나리오 ID | 유형 | 기대 |
|-------------|------|------|
| TP-01 | 정상: 시도 1 성공 | `int[6]` 고정 기대 |
| TP-02 | 정상: 시도 1 실패, 시도 2 성공 | 역순 규칙 반영 |
| TP-03 | 실패: 두 시도 모두 실패 | `E_DOMAIN` |
| TP-04 | 입력 오류: 크기 | `E_SIZE` |
| TP-05 | 입력 오류: 빈칸 개수 | `E_EMPTY_COUNT` |
| TP-06 | 입력 오류: 값 범위 | `E_VALUE_RANGE` |
| TP-07 | 입력 오류: 중복 | `E_DUPLICATE` |

### 9.2 회귀 테스트 정책

- 공개 계약(`int[6]` 순서, 오류 코드·메시지, INV ID) 변경 시 **문서 버전 상향** 및 추적 매트릭스 갱신.
- 기존 테스트 삭제 대신 **기대값 변경 시** 커밋 메시지에 근거 기재.
- Golden 파일을 사용하는 경우 저장소에 **결정적** 입력만 둔다.

### 9.3 테스트 데이터 — 대표 4×4 행렬

**시드 A (Report/4 Gherkin 정렬, 시도 1 성공 경로 검증용 소재)**

| 열1 | 열2 | 열3 | 열4 |
|-----|-----|-----|-----|
| 16 | 2 | 3 | 13 |
| 5 | 11 | 10 | 8 |
| 9 | 7 | 0 | 12 |
| 4 | 14 | 15 | 0 |

- 1-index 첫 빈칸 `(3,3)`, 둘째 `(4,4)`.
- 누락 숫자: `1`, `6` (`n_small=1`, `n_large=6`).
- 구현 확정 후 기대 `int[6]`은 단위 테스트에서 상수로 고정한다.

**시드 B (시도 1 실패·시도 2 성공)**

- 별도 행렬을 테스트 코드 옆에 **주석 또는 픽스처 파일**로 명시하고, 기대 `int[6]`을 함께 고정한다(본 PRD는 값만 요구하지 않고 존재를 요구).

**시드 C (두 시도 모두 실패)**

- 유효 입력(INV 만족)이면서 두 배치 모두 비마방진인 예를 1건 이상 확보한다.

### 9.4 Property / Invariant 기반 체크

- 완성 그리드에서 1~16이 한 번씩 등장하는지.
- 성공 시 네 행·네 열·두 대각선 합이 34인지 자동 검증 헬퍼로 재검증.

## 10. Architecture Overview (High-Level)

### 10.1 레이어

| 레이어 | 책임 |
|--------|------|
| **Boundary** | `int[][]` 수신, FR-01 검증, 성공 시 `int[6]` 또는 표준 오류 구조 반환. 프레임워크·콘솔·테스트 하네스는 동일 계약으로 호출 가능. |
| **Domain** | FR-02~FR-05: 빈칸·누락 수·완성 판정·두 시도. 외부 I/O 없음. |
| **Control (선택)** | Boundary와 Domain 사이 유스케이스 조합만 담고, 도메인 규칙을 복제하지 않는다. |

### 10.2 책임 분리 및 확장

- **SRP:** 검증·탐색·판정·순서 결정을 파일/타입 단위로 분리 가능하나 본 PRD는 인터페이스 이름을 강제하지 않는다.
- **OCP:** N×N 확장은 새 타입 또는 새 전략으로 **추가**할 여지를 두되, 기본 배포는 4×4 고정 동작이다.

### 10.3 의존성 방향

- Boundary → Domain (또는 Boundary → Control → Domain).
- Domain은 Boundary를 **참조하지 않는다**.

## 11. Risks & Ambiguities

| 결정 항목 | 고정 내용 |
|-----------|-----------|
| 두 시도 모두 실패 | `E_DOMAIN` + 메시지 `No valid magic square completion exists for this grid.` |
| 좌표 체계 | 1-index, 행·열 모두 `1..4` |
| 대각선 | 주 대각선·반대 대각선 각각 합 34 |
| 입력 변경 | NFR-04: 호출자 배열 불변 |
| “UI” 용어 | 본 문서에서 Boundary 계약층을 의미하며 웹 UI가 아니다. |

**자주 실수하는 포인트**

- 0-index와 1-index 혼동.
- row-major 이외의 빈칸 순서 사용.
- 출력을 `[r1,c1,r2,c2,n1,n2]`가 아닌 다른 순열로 반환.
- 검증 실패 시에도 도메인 호출.
